## What landed

`MainWindow::showDiffViewer` (~430 LoC of dialog construction +
five async git probes + `QFileSystemWatcher` + scroll-preservation
+ Copy-Diff payload) moved to a free function
`diffviewer::show(QWidget *parent, const QString &cwd, const
QString &themeName)` in new translation unit
`src/diffviewer.{cpp,h}`.

`mainwindow.cpp` shrank from 6636 → 6213 LoC. The post-extraction
`showDiffViewer` slot is ~25 meaningful LoC: status message →
focused-terminal lookup → button disable → call into
`diffviewer::show` → connect dialog `destroyed` for re-enable +
`refreshReviewButton`.

## Process notes

The user invoked the strict App-Build spec → cold-eyes review →
fold-into-spec → fix-pass loop. First-pass spec was technically
adequate but silently chose the surface design and under-specified
the per-test re-pointing work for the four pre-existing
`review_changes_*` feature tests. Cold-eyes review (general-
purpose subagent, briefed only with spec + pre-extraction
mainwindow.cpp + project standards) returned 1 CRITICAL + 2 HIGH +
3 MEDIUM + 1 LOW finding. All folded into the spec before the
implementation was reviewed:

- **CRITICAL** INV-5 reframed as a per-test table — only one of
  the four review_changes_* tests survives a literal define-
  rename; two need source edits (search-string change from
  `void MainWindow::showDiffViewer()` to `QDialog *show(`); the
  fourth needs no change at all (it was about
  `refreshReviewButton`, not the dialog body).
- **HIGH** Surface-design choice (`QDialog*` return + caller-
  wires-destroyed vs 5-arg callback form) is now justified
  explicitly in the spec — the dialog's responsibility is
  rendering diff state, not managing an arbitrary external
  button; Rule of Three says don't pre-DRY for a hypothetical
  second caller.
- **HIGH** Added INVs pinning the exact function signature and
  every user-visible string (window title, button labels, live-
  status texts, section headers, empty-state line) byte-
  identical — guards against drive-by reformatting.
- **MEDIUM** Theme-capture-at-open and QProcess-parent-change
  semantics named explicitly so a future reader doesn't "fix"
  them.
- **MEDIUM** LoC cap on the post-extraction body relaxed from
  ≤30 to ≤50 with explicit counting algorithm (physical lines
  between `{` and `}`, exclusive of blanks and lines starting
  with `//`).

This loop also surfaced a process-level lesson the user named
explicitly: **think the problem through before writing a spec or
code, not after**. The cold-eyes reviewer should be a check on
the up-front thinking, not the source of design. Saved as
feedback memory `feedback_think_before_spec.md` for future runs.
