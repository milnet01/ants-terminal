# ANTS-1159 — Crash-safe session persistence

> **Status:** spec — pre-implementation
> **Source:** user report 2026-05-02 ("after the crash, none of my
> tabs persisted though"). Approved option B (periodic timer +
> tab-event saveTabOrder) at design surface.
>
> **Lanes:** `MainWindow` (signal hooks, `m_sessionSaveTimer`).
> No SessionManager API change — the existing
> `saveSession` / `saveTabOrder` are reused.
> **Kind:** fix

## 1. Problem

`MainWindow::saveAllSessions()` is invoked **only** from
`MainWindow::closeEvent` (mainwindow.cpp:3798). Any abnormal
termination — SIGSEGV, OOM-kill, power loss, kernel panic —
bypasses `closeEvent` and discards everything written since the
last graceful shutdown:

- New tab IDs created since last close → never persisted.
- Tab-order file (`tab_order.txt`) → never refreshed.
- Per-tab scrollback `.dat` blobs → never refreshed.

User-visible symptom: after a crash, the next ants-terminal
launch sees zero saved sessions and falls back to opening one
default tab. The user lost the entire workspace.

ANTS-1141 already addressed a narrower variant (the destructive
read on `tab_order.txt` that lost ordering after crash within the
5-s uptime floor). This spec addresses the broader case.

## 2. Goal

Bound the worst-case loss window after a crash to:

- **Tab list (which tabs exist, in what order)**: ≤ event-loop
  latency. Saved on every tab create / close / reorder, and on
  initial launch once `m_uptimeTimer.elapsed() >= 5000`.
- **Per-tab session content (scrollback, cwd, pin)**: ≤ 30 s.
  Saved by a periodic timer + on graceful close.

Tab list is the cheap, high-value subset. Scrollback is the
heavier blob — bounded but not zero-loss.

## 3. Non-goals

- **Zero-loss persistence.** Would require save-on-every-keystroke
  with hot-path debouncing — out of scope for this fix-pass.
- **Crash-handler save.** `SIGSEGV` handler context limits us to
  async-signal-safe operations (no QFile, no QString allocation,
  no Qt event-loop). The cost/benefit of pre-allocating an
  emergency-save scratch buffer doesn't pay off when periodic
  saves cap loss to 30 s.
- **Distinguishing user-initiated vs auto-save.** All saves are
  equivalent on disk (atomic-rename of `<file>.tmp` → `<file>`).
- **Mid-shutdown handling.** `closeEvent` already calls
  `saveAllSessions`. The new periodic timer must not stomp it on
  the way out — handled by stopping the timer in `closeEvent`
  before the manual save fires.

## 4. Design

### 4.1 Periodic full save

- New `QTimer *m_sessionSaveTimer` member on `MainWindow`.
- Interval: **30 000 ms (30 s)**. Single-shot OFF; it autoreloads.
- Slot: existing `saveAllSessions()`. The 5-second uptime floor
  inside `saveAllSessions` already guards against startup-time
  no-op runs.
- Started in the `MainWindow` constructor after `restoreSessions()`
  completes (so the first save isn't fighting the restore).
- Stopped in `closeEvent` *before* the explicit `saveAllSessions()`
  call, so the timer can't re-enter mid-shutdown.

### 4.2 Tab-event tab-order save

- Three signal hookups:
  - `QTabWidget::tabCloseRequested` is **already wired** to
    `MainWindow::closeTab`. After `performTabClose` removes the
    tab, call `saveTabOrderOnly()`.
  - `QTabBar::tabMoved(int from, int to)` — connect on the
    `m_coloredTabBar` (not the QTabWidget) → `saveTabOrderOnly()`.
  - `MainWindow::newTab()` and `MainWindow::newTabWith*` paths
    (and the restore-default path) — after the tab is added,
    call `saveTabOrderOnly()`.
- `saveTabOrderOnly()` is a **new private slot** that does the
  cheap subset of `saveAllSessions`: walks `m_tabWidget`, builds
  the `tabOrder` QStringList + active index, calls
  `SessionManager::saveTabOrder(...)`. Skipped when
  `m_config.sessionPersistence()` is off **or** when
  `m_uptimeTimer.elapsed() < 5000` (same uptime floor as
  `saveAllSessions` — an early-launch crash shouldn't wipe the
  on-disk order with a half-restored list).
- `currentChanged` (tab switch) is **NOT** hooked. Tab switches
  fire constantly during normal use and only update
  `activeIndex`; the periodic 30 s timer is sufficient.

### 4.3 Interaction with auto-profile + session-persistence-off

- When `m_config.sessionPersistence()` is OFF, both
  `saveAllSessions` and `saveTabOrderOnly` early-return. The
  timer can keep running (idle no-op) — simpler than gating
  start/stop on the config bit.
- The auto-profile path that calls `applyTheme` doesn't touch
  the session save path. No interaction.

## 5. Invariants (for `tests/features/crash_safe_session_persist/`)

Source-grep based, mirroring
`tests/features/ui_state_persistence/spec.md`:

- **INV-1** — `m_sessionSaveTimer` member exists on `MainWindow`
  and is initialised with interval `30000`.
- **INV-2** — Timer's `timeout` signal is connected to
  `saveAllSessions`.
- **INV-3** — `saveTabOrderOnly` (or equivalently-named cheap
  tab-order-only save) is defined as a member.
- **INV-4** — `saveTabOrderOnly` is called from `newTab`
  (post-`addTab`).
- **INV-5** — `saveTabOrderOnly` is called from `performTabClose`
  (post-`removeTab`).
- **INV-6** — `saveTabOrderOnly` is connected to
  `m_coloredTabBar`'s `tabMoved` (or `m_tabWidget->tabBar()`'s
  `tabMoved`).
- **INV-7** — `closeEvent` stops `m_sessionSaveTimer` before
  calling `saveAllSessions` (no re-entry).
- **INV-8** — `saveAllSessions` is **still** called from
  `closeEvent` (no regression of the existing save path).
- **INV-9** — `saveTabOrderOnly` short-circuits on
  `!sessionPersistence` and on the 5-second uptime floor (same
  guards as `saveAllSessions`).

## 7. Acceptance

Spec is met when:

1. `saveAllSessions` is observed to fire on a 30 s wall-clock
   cadence in normal use (manual `kill -USR1` would be ideal,
   but timer-fire suffices).
2. Creating a new tab (Ctrl+T) writes `tab_order.txt` within one
   event-loop turn (verified by `inotifywait` on the sessions
   directory while issuing Ctrl+T).
3. Closing a tab (× button or Ctrl+W) writes `tab_order.txt`.
4. Drag-reordering tabs writes `tab_order.txt` after the drop.
5. Issuing `kill -SEGV $(pidof ants-terminal)` and immediately
   restarting recovers all tabs that existed > 5 s before the
   kill, in their last-known order.
6. All INVs in `tests/features/crash_safe_session_persist/`
   pass.

## 8. Risk + rollback

- **Risk:** double-save during graceful shutdown if `closeEvent`
  fires between two timer ticks. Mitigated by stopping the
  timer in `closeEvent` before the manual `saveAllSessions`
  call.
- **Risk:** disk wear from periodic writes. 30 s × 24 h ≈ 2880
  saves/day; each save is one atomic rename per tab + one for
  tab_order.txt. SSDs handle this trivially. Rotational disks
  on a long-running session might hit ~5 K writes/day, still
  well below disk-wear thresholds.
- **Rollback:** revert the MainWindow diff. The session API is
  unchanged; on-disk format is unchanged. No migration needed.

## 9. What this does NOT do (defer for separate IDs if a user asks)

- Per-grid mutation debouncing for sub-30-s scrollback
  durability.
- Crash-handler-driven async-signal-safe emergency save.
- Cross-instance lock (multiple ants-terminal processes saving
  to the same `~/.local/share/ants-terminal/sessions/` — already
  handled by atomic rename, but contention on the same tab IDs
  could produce write-amplification; out of scope here).
