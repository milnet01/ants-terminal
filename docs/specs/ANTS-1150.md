# ANTS-1150 — UI / chrome state persistence (Phase 1)

- **Status:** Spec — cold-eyes review folded in 2026-05-02 (2
  CRITICAL, 5 HIGH, 6 MEDIUM, 4 LOW)
- **Source:** user-2026-05-01 ("Everything that makes sense to
  persist across sessions, please make that persist.")
- **Lanes:** Config, SettingsDialog, RoadmapDialog, AuditDialog
- **Kind:** implement (new persistence keys + load/save hooks; no
  behavioural change to the underlying dialog logic)
- **Out of scope (Phase 2):** RoadmapDialog splitter position,
  AuditDialog filter-bar splitter, RoadmapDialog search-box content
  + per-tab terminal search-bar content + last-search direction,
  AuditDialog last-selected finding, AuditDialog confidence-sort
  toggle, AuditDialog recent-only / recent-lines-only toggles,
  SettingsDialog last-active sub-state inside any tab,
  CommandPalette recent-actions cache. Splitters are deferred
  because they require base64 `saveState/restoreState` blobs
  (cheap, but adds JSON shape) and the user signed off on the
  "obvious-yes" Phase 1 only.

## Problem

Today's restart UX has uneven state retention. Some bits stick
(window geometry, theme, opacity, tab session content, tab order,
RoadmapDialog window geometry, repo-visibility cache, latest known
remote version) — see `Config::*Geometry`, `SessionManager::*`,
the existing `roadmap_dialog_geometry` key. Other equally-frequent
choices reset on every launch:

1. **SettingsDialog tab.** Open Settings, click Appearance, tweak,
   close. Re-open Settings — back to General. The tab choice is
   throwaway state that the user re-makes every visit.

2. **RoadmapDialog active preset.** User switches to "Current" or
   "Next" (per their typical workflow), closes the dialog. Next
   open — back to "Full roadmap" by default
   (`roadmapdialog.cpp:1025`: `applyPreset(Preset::Full)`).

3. **RoadmapDialog Kind facets.** Per ANTS-1106 spec the Kind-facet
   checkbox set was explicitly chosen to reset on each open. User
   ask 2026-05-01 reverses that decision — power users who slice
   the roadmap by `audit-fix` or `fix` want the slice to survive
   a restart.

4. **RoadmapDialog 5 status-emoji checkboxes** (Done / Planned /
   In progress / Considered / Currently being tackled). For named
   presets these are derived from the preset; for Custom they're
   user-tuned. Today: lost on close.

5. **AuditDialog severity-filter chips** (BLK / CRT / MAJ / MIN /
   INF). The user toggled MIN + INF off because they were drowning
   in noise. Re-open the dialog after a re-run — all five back on.
   Has to be re-toggled every audit session.

6. **AuditDialog "Show new since baseline" toggle.** Once a user
   has saved a baseline, this is the most common review mode.
   Resets to off on every re-open.

## Discovery during exploration (deviations from the meta-bullet's
candidate list)

The 2026-05-01 meta-bullet listed **AuditDialog "Source-filter

group in `auditdialog.cpp` — actual filter widgets are: text

confidence-sort toggle (`m_confidenceSortBtn`), show-new-only
toggle (`m_newOnlyBtn`), recent-only / recent-lines-only toggles.
Dropped from Phase 1; substitution candidates (confidence-sort,

The meta-bullet also listed **active terminal tab index across

below.

## Fix

### Cohort

1. **SettingsDialog last-active tab** — `settings_dialog_last_tab`
   (int; tab index 0..8). Restored on dialog ctor after
   `loadSettings()`; saved on `m_tabs->currentChanged(int)`

2. **RoadmapDialog active preset** — `roadmap_active_preset`
   (string; one of `"full"`, `"history"`, `"current"`, `"next"`,
   `"far_future"`, `"custom"`). Restored at end of ctor (replaces
   the unconditional `applyPreset(Preset::Full)` at
   `roadmapdialog.cpp:1025`); saved inside `applyPreset(Preset)`
   for named-preset transitions and inside `onCheckboxToggled()`
   when the preset flips to Custom.
3. **RoadmapDialog Kind facet checkbox set** —
   `roadmap_kind_filters` (array of strings; values from the
   `KindEntry` table at `roadmapdialog.cpp:846` — `"implement"`,
   `"fix"`, `"audit-fix"`, `"review-fix"`, `"doc"`, `"doc-fix"`,
   `"refactor"`, `"test"`, `"chore"`, `"release"`, `"research"`,
   `"ux"`). Restored in ctor before `applyPreset` (so the kind
   checkboxes reflect the persisted set when the dialog first

4. **RoadmapDialog 5 status-emoji checkboxes** —
   `roadmap_status_filters` (object: `{ "done": bool, "planned":

5. **AuditDialog severity-filter chips** — `audit_severity_filters`

### Config plumbing

Six new public getters and six new setters on `Config`. All
follow the established `storeIfChanged` + `save()` short-circuit
pattern from `Config::setTheme` etc. Sample shape:

```cpp
// config.h
int settingsDialogLastTab() const;
void setSettingsDialogLastTab(int index);

QString roadmapActivePreset() const;
void setRoadmapActivePreset(const QString &name);

QStringList roadmapKindFilters() const;
void setRoadmapKindFilters(const QStringList &kinds);

QJsonObject roadmapStatusFilters() const;
void setRoadmapStatusFilters(const QJsonObject &filters);

QJsonObject auditSeverityFilters() const;
void setAuditSeverityFilters(const QJsonObject &filters);

bool auditShowNewOnly() const;
void setAuditShowNewOnly(bool enabled);
```

Defaults on first read (no persisted key yet):

| Key | Default |
|---|---|
| `settings_dialog_last_tab` | `0` (General) |
| `roadmap_active_preset` | `"full"` |
| `roadmap_kind_filters` | `[]` (empty — show all kinds, matches today's "Kind filter resets on each open" behaviour for users who never tune it) |
| `roadmap_status_filters` | `{}` (empty object → falls through to per-preset default in `applyPreset`, matches today's first-launch behaviour) |
| `audit_severity_filters` | `{}` (empty → all five severities on, matches the `m_activeSeverities = {0,1,2,3,4}` default at `auditdialog.h:467`) |
| `audit_show_new_only` | `false` |

Validation in getters:
- `settingsDialogLastTab()` — clamps the read int to
  `[0, max-tab-index]` *at the call site* (the dialog ctor),
  not in the getter, because `Config` doesn't know how many
  tabs SettingsDialog has. Mirrors the
  `restoreSessions:3308` clamp pattern.
- `roadmapActivePreset()` — returns `"full"` if the persisted
  string is not in the known set. RoadmapDialog ctor then maps
  string → enum; unknown string → `Preset::Full`.
- `roadmapKindFilters()` — silently drops persisted strings that
  aren't in the known kind set (defends against a user editing
  config.json by hand and a stale value lingering after a
  KindEntry rename).
- `roadmapStatusFilters()` — returns whatever object is stored
  (empty → caller falls back to preset-derived defaults).
- `auditSeverityFilters()` — same shape; empty object means
  "all on."
- `auditShowNewOnly()` — defaults to `false`.

### Save / load wiring per dialog

**SettingsDialog** (`settingsdialog.cpp`):
- After `loadSettings()` at line 86: read `m_config->settingsDialogLastTab()`,
  clamp to `[0, m_tabs->count()-1]`, call `m_tabs->setCurrentIndex(clamped)`
  **wrapped in `QSignalBlocker block(m_tabs);`** so the restore
  doesn't write back through the just-installed `currentChanged`
  handler. Without the blocker, the restore writes the clamped
  value back to disk on every open — usually harmless but racy
  if two windows clamp to different limits (older window with 9
  tabs persists 5, newer window with 4 tabs reads 5 → clamps to 3
  → writes 3 → older window's later open reads 3 instead of 5).
- Add `connect(m_tabs, &QTabWidget::currentChanged, this,
  [this](int idx) { if (m_config) m_config->setSettingsDialogLastTab(idx); });`
  before the existing connects in the ctor.

**RoadmapDialog** (`roadmapdialog.cpp`):

**Order matters.** The ctor restore runs BEFORE the existing
`applyPreset(Preset::Full)` at line 1025. CRITICAL: restoring
status checkboxes silently and then calling `applyPreset(Full)`
would have applyPreset's mask-application block at line 1072-1085
overwrite every just-restored checkbox with the named-preset's
canonical mask — net effect, persisted status filters only matter
when the persisted preset is Custom. Sequence:

1. Restore Kind filter set (always — Kind is preset-orthogonal).
2. Determine persisted preset string → enum.
3. **Only if persisted == Custom**, restore status checkboxes
   silently. For named presets, applyPreset will write the
   canonical mask anyway, so the restore is dead code.
4. **Custom + empty status filters edge case:** if the persisted
   preset is `"custom"` AND `roadmap_status_filters` is empty/
   missing (e.g. hand-edited config, or ANTS-1106 spec migration),
   fall back to `Preset::Full` to avoid an empty-render dead-end.

**Two write sites for `m_activePreset`** — `applyPreset` (named-
preset path) and `onCheckboxToggled` (Custom-fallback path at
line 1119-1127). Both must persist or restore will silently regress
on every Custom-via-divergence flip. Spec uses a private helper
`persistActivePreset(Preset)` shared by both sites and a `switch`
statement (compiler-checked enum coverage; mirrors `filterFor` at
line 267-284) instead of a name array indexed by `int(Preset)` —
inserting a new Preset value before Custom would silently shift
indices.

**Lift the KindEntry table to file scope** so the build loop at
line 846 and the new restore loop both iterate the same table —
no name-prefix `findChildren` walk, no startsWith string magic,
no implicit naming-convention coupling. Cache the QCheckBox
pointers in a `QHash<QString, QCheckBox *> m_kindCheckboxes`
member during build for O(1) lookup during restore and for the
existing toggled-lambda reuse.

Replace the unconditional `applyPreset(Preset::Full)` at line
1025 with:

## Risks / non-risks

- **No risk of clobbering an in-progress edit.** Every write goes
  through `storeIfChanged` → `save()`, which holds the
  `ConfigWriteLock` and uses the existing atomic-write +
  `m_loadFailed` suppression. If two Ants instances write
  simultaneously, the lock arbitrates — last-saver-wins per key
  is the documented behaviour and the user-facing UX is "your
  preset choice survives, except possibly when two windows toggle
  it within the same lock window."
- **No risk of inotify-loop regression.** `MainWindow`'s existing
  `onConfigFileChanged` handler relies on the
  `storeIfChanged` short-circuit at the setter level — adding
  new keys that follow the same pattern doesn't break that
  invariant. INV-7 locks it in for the new keys.
- **No risk of the persisted RoadmapDialog state masking a
  future Kind enum addition.** `roadmapKindFilters()` drops
  unknown strings on read (validation step in the getter
  implementation). New entries in the `KindEntry` table at
  `roadmapdialog.cpp:846` start unchecked, which is the same as
  today's first-launch behaviour.
- **Risk: SettingsDialog tab reorder.** If a future PR reorders
  the 9 SettingsDialog tabs, the saved index points at the wrong
  tab on the next launch. Acceptable: low frequency, recoverable
  in one click. Not enough to justify a stable-string-ID scheme.
- **Corrupt-config interaction.** When `Config::loadFailed()` is
  true, `save()` is suppressed for the rest of the session
  (`config.cpp:99-105`). All persisted-state restores fall through
  to defaults (because `m_data` is empty), and any UI toggles
  during that session evaporate at next launch — but the rotated-
  aside `config.json.corrupt-<timestamp>` is preserved for
  hand-recovery. Same UX as every other persisted Config key
  today.
- **Hand-edit of config.json with wrong types defaults to ON.**
  `QJsonValue::toBool(defaultValue)` returns the default for
  *both* missing keys *and* type-mismatched values (a string
  `"false"`, a number `0`, etc.). All status/severity restores in
  this spec use `.toBool(true)` defaults, so a hand-edit with the
  wrong JSON type silently restores to ON. Acceptable for
  checkbox-style filters (default-on matches first-launch
  behaviour); document for users editing config.json — bare
  `true`/`false` literals only.
