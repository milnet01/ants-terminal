# ANTS-1146 — Extract ClaudeStatusBarController from mainwindow.cpp

- **Status:** Implementing (target 0.7.74, bundled with ANTS-1147)
- **Source:** indie-review-2026-05-01 (L6 LoC2) + brainstorm 2026-05-02
- **Lanes:** mainwindow, claudestatuswidgets (new)
- **Kind:** refactor (no behavioural change)

## Problem

`mainwindow.cpp` is 6249 lines. Three contiguous Claude-status
sections account for ~680 of those:

- `MainWindow::applyClaudeStatusLabel` — 3561-3669 (~108 LoC)
- `MainWindow::updateClaudeThemeColors` — 3669-3731 (~62 LoC)
- `MainWindow::setupClaudeIntegration` — 3731-4239 (~510 LoC)

`setupClaudeIntegration` is the second-largest function in the
file (after `showDiffViewer` was carved out by ANTS-1145). It
owns four widgets, a permission-button-group factory (~120 LoC),
a tab-indicator provider, six `ClaudeIntegration` signal connect
blocks, the `ClaudeBgTaskTracker` + button, and three orphans
that are *not* Claude-state at all (the Roadmap button, the
update-available menu action, and the 5-second startup
update-check `singleShot`) — they landed in this function for
historical convenience.

Splitting the Claude-specific portion into its own translation
unit (a) shrinks `mainwindow.cpp` by ≥ 480 LoC toward the
sub-2000-LoC target, (b) gives the Claude status-bar widgets a
clear test surface (one `.cpp` to grep, not three sections of a
6249-line file), (c) sets up the theming carve-out (ANTS-1147)
to land cleanly on a smaller `mainwindow.cpp`, and (d) is the
natural sibling to ANTS-1145's `DiffViewerDialog` carve-out
under the same indie-review L6 banner.

The three orphans (Roadmap button, update-available action, 5 s
update-check timer) stay in MainWindow — they are not Claude
chrome and would mis-name a "Claude" controller. Their host
function gets renamed `setupClaudeIntegration` → `setupStatusBarChrome`.

## Fix

New translation unit `src/claudestatuswidgets.{cpp,h}` exposing
one class at the top level (no namespace — matches the codebase
convention used by `ClaudeIntegration`, `ClaudeTabTracker`,
`ClaudeAllowlistDialog`, `DiffViewerDialog`, etc.; sidesteps the
PMF-qualifier story in `mainwindow.cpp` connect calls):

```cpp
class ClaudeStatusBarController : public QObject {
    Q_OBJECT
public:
    explicit ClaudeStatusBarController(QStatusBar *statusBar,
                                       QObject *parent);

    // ── Wiring (called once from MainWindow ctor) ──
    void attach(ClaudeIntegration *integration,
                ClaudeTabTracker  *tracker,
                ColoredTabBar     *coloredTabBar,
                QTabWidget        *tabWidget);

    // ── State pokes (MANUAL — from MainWindow's terminal-event
    //    handlers; auto state from ClaudeIntegration::stateChanged /
    //    planModeChanged / auditingChanged is consumed by the
    //    controller's own connect blocks set up in attach()). ──
    void setPromptActive(bool active);
    void setPlanMode(bool active);
    void setAuditing(bool active);

    // ── Error-label surface (used by MainWindow's commandFailed
    //    handler to surface "Exit N" with output tooltip). ──
    void setError(const QString &text,
                  const QString &tooltip,
                  int autoHideMs);
    void clearError();

    // ── Tab-switch reset, called from onTabChanged. Bundles the
    //    five-line clear-and-hide block at mainwindow.cpp:4604-4612
    //    that today directly mutates m_claudeLastState,
    //    m_claudeLastDetail, m_claudePlanMode, m_claudeAuditing,
    //    hides m_claudeReviewBtn / m_claudeContextBar /
    //    m_claudeBgTasksBtn, and resets the bg-tasks transcript
    //    path. Single entry point keeps the reset atomic. ──
    void resetForTabSwitch();

    // ── Theme application (called from MainWindow's central
    //    restyle helper — this REPLACES updateClaudeThemeColors). ──
    void applyTheme(const QString &themeName);

    // ── External refresh entry-points called from MainWindow's
    //    tab/timer plumbing (status timer pulse, onTabChanged,
    //    showBgTasksDialog dismissal). Body reads focusedTerminal()
    //    + m_claudeIntegration (both already supplied) + internal
    //    tracker state. ──
    void refreshBgTasksButton();

    // ── Provider injection (Qt-idiomatic; matches the existing
    //    ClaudeIntegration::set*Provider pattern). ──
    void setCurrentTerminalProvider(std::function<TerminalWidget *()>);
    void setFocusedTerminalProvider(std::function<TerminalWidget *()>);
    void setTerminalAtTabProvider(std::function<TerminalWidget *(int)>);
    // The tab-indicator provider lambda reads the user-toggle
    // m_config.claudeTabStatusIndicator() which lives on Config —
    // injected as a callback so the controller doesn't drag in
    // the Config header.
    void setTabIndicatorEnabledProvider(std::function<bool()>);

    // ── Accessors (inline non-virtual; legacy MainWindow paths
    //    such as refreshReviewButton + showDiffViewer use these
    //    rather than re-acquiring direct member references). ──
    QPushButton *reviewButton() const;
    QPushButton *bgTasksButton() const;
    ClaudeBgTaskTracker *bgTasksTracker() const;

signals:
    void reviewClicked();
    void bgTasksClicked();
    void allowlistRequested(const QString &rule);
    void reviewButtonShouldRefresh();
    void statusMessageRequested(const QString &text, int timeoutMs);
    void statusMessageCleared();
};
```

The controller member on MainWindow is named
`m_claudeStatusBarController` (not `m_claudeStatusBar` — the
latter would collide cognitively with `statusBar()` which
returns the `QStatusBar*`).

### Surface-design choice (justified)

Three honest options were enumerated in the brainstorm:

- **(A)** Big controller class with widgets + state + wiring
  + provider callbacks + signals out. ≈500 LoC moves; matches
  the roadmap entry's intent; mirrors how Qt-style controllers
  are typically organised in this codebase.
- **(B)** Free function returning a struct of widgets + pure
  apply / theme functions; state stays on MainWindow. ≈150
  LoC moves; simpler but doesn't hit the carve-out target and
  leaves the wiring half on MainWindow.
- **(C)** Hybrid: controller owns widgets + theme; state booleans
  + signal wiring stay on MainWindow. ≈300 LoC; compromise.

**Decision: (A).** Reasoning:

- The state booleans (`promptActive`, `planMode`, `auditing`,
  `lastState`, `lastDetail`) are mutated by exactly two
  consumers — the status-bar render path and a handful of
  terminal-event handlers in MainWindow. (B)/(C) leave them
  on MainWindow; the only call site in MainWindow that sets
  them then immediately calls `apply()` (now `controller->...`).
  Moving them onto the controller eliminates the cross-class
  state coupling without adding new state-ownership ambiguity.
- The permission-button-group factory (~120 LoC at lines
  4010-4172) is the largest connected sub-section of
  `setupClaudeIntegration`. It belongs with the rest of the
  Claude status-bar chrome — leaving it on MainWindow under
  (B)/(C) would split a tightly cohesive block across two
  TUs for no semantic gain.
- (B) reads better than (A) for one-shot dialogs (the 1145
  case); (A) reads better than (B) for long-lived controllers
  with stateful widgets (the 1146 case).

### Coupling shape: signals out + std::function providers in

The controller talks back to MainWindow via two channels:

1. **Signals** (Qt-idiomatic): `reviewClicked`,
   `bgTasksClicked`, `allowlistRequested(rule)`,
   `reviewButtonShouldRefresh`, `statusMessageRequested(text,
   ms)`, `statusMessageCleared`. MainWindow's existing slots
   (`showDiffViewer`, `showBgTasksDialog`,
   `openClaudeAllowlistDialog`, `refreshReviewButton`,
   `showStatusMessage`, `clearStatusMessage`) connect once at
   ctor time.
2. **`std::function` providers**: `setCurrentTerminalProvider`,
   `setFocusedTerminalProvider`, `setTerminalAtTabProvider`.
   Pattern matches the existing
   `ClaudeIntegration::setScrollbackProvider` /
   `setCwdProvider` etc — codebase reads consistently.

A single abstract `Bridge` interface (option Q2/(i)) was
considered and rejected: there is one consumer, no plans for
a second; YAGNI applies.

### Setup choreography in MainWindow

```cpp
// Replaces the m_claudeIntegration = new ClaudeIntegration(this)
// + setupClaudeIntegration() pair in MainWindow ctor.
m_claudeIntegration = new ClaudeIntegration(this);
m_claudeTabTracker  = new ClaudeTabTracker(this);

m_claudeStatusBarController =
    new ClaudeStatusBarController(statusBar(), this);
m_claudeStatusBarController->setCurrentTerminalProvider(
    [this]{ return currentTerminal(); });
m_claudeStatusBarController->setFocusedTerminalProvider(
    [this]{ return focusedTerminal(); });
m_claudeStatusBarController->setTerminalAtTabProvider(
    [this](int i){ return terminalAtTab(i); });
m_claudeStatusBarController->setTabIndicatorEnabledProvider(
    [this]{ return m_config.claudeTabStatusIndicator(); });
m_claudeStatusBarController->attach(
    m_claudeIntegration, m_claudeTabTracker,
    m_coloredTabBar, m_tabWidget);
m_claudeStatusBarController->applyTheme(m_currentTheme);

connect(m_claudeStatusBarController, &ClaudeStatusBarController::reviewClicked,
        this, &MainWindow::showDiffViewer);
connect(m_claudeStatusBarController, &ClaudeStatusBarController::bgTasksClicked,
        this, &MainWindow::showBgTasksDialog);
connect(m_claudeStatusBarController, &ClaudeStatusBarController::allowlistRequested,
        this, &MainWindow::openClaudeAllowlistDialog);
connect(m_claudeStatusBarController, &ClaudeStatusBarController::reviewButtonShouldRefresh,
        this, &MainWindow::refreshReviewButton);
connect(m_claudeStatusBarController, &ClaudeStatusBarController::statusMessageRequested,
        this, [this](const QString &t, int ms){ showStatusMessage(t, ms); });
connect(m_claudeStatusBarController, &ClaudeStatusBarController::statusMessageCleared,
        this, &MainWindow::clearStatusMessage);

setupStatusBarChrome();   // renamed from setupClaudeIntegration —
                          // constructs the three orphans only.
```

### Behavioural parity (explicit)

Every connect block, the permission-button factory's
`claudeAllowBtn` `objectName` (so `onTabChanged`'s cleanup
walk keeps finding both flavours), the SGR colour translation
in `apply` (preserved by INV-7), the tab-indicator provider
lambda's nine glyph branches, the context-bar 60/80 % colour
threshold logic, the tooltip texts, the accessible names,
the dedup `findChildren` walk, the proxy-signal retraction
(`toolFinished` / `sessionStopped`), the prompt-active
auto-relabel, the per-tab tab-glyph orange flag — all move
byte-for-byte. The only mechanical changes:

- `applyClaudeStatusLabel()` becomes `apply()` (private method)
- `updateClaudeThemeColors()` becomes `applyTheme(QString)` (public)
- `MainWindow::refreshBgTasksButton` moves wholesale to
  `ClaudeStatusBarController::refreshBgTasksButton` (its body
  uses controller-side tracker + button state plus the
  already-supplied `focusedTerminal` provider and
  `m_claudeIntegration` pointer — no MainWindow-only deps).
  Three MainWindow external call sites swap from
  `refreshBgTasksButton()` to `m_claudeStatusBarController->refreshBgTasksButton()`:
  - `mainwindow.cpp:681` — `connect(m_statusTimer, ..., m_claudeStatusBarController, &ClaudeStatusBarController::refreshBgTasksButton)`
  - `mainwindow.cpp:4642` — call inside `onTabChanged`
  - `mainwindow.cpp:5323` — call inside `showBgTasksDialog` after dialog dismissal
- `MainWindow::refreshReviewButton` **stays on MainWindow** —
  its body reads MainWindow-only state (`m_reviewProbeInFlight`
  + the git-probe `QProcess` machinery) that has no controller
  counterpart. The two pre-existing `m_claudeReviewBtn`
  references inside it (and the one in `onTabChanged`) swap
  to `m_claudeStatusBarController->reviewButton()->...`.
- `m_claude<X>` direct member references become `this->m_<X>`
  (within the controller) or `m_claudeStatusBarController-><method>`
  (from MainWindow's terminal-event handlers, ~8 sites)
- MainWindow's `showDiffViewer`, `openClaudeAllowlistDialog`,
  `currentTerminal`, `focusedTerminal`, `terminalAtTab`,
  `showStatusMessage`, `clearStatusMessage` calls inside the
  carved-out body become signal emits / provider calls per the
  table above.

Two semantics named explicitly so a future reader doesn't
"fix" them:

- **Controller holds the current theme name as a private
  member** (`QString m_currentThemeName`) updated by
  `applyTheme(QString)`. The `contextUpdated` connect-block
  reads it to choose the green/yellow/red chunk colour on
  every percentage tick (today the lambda reads
  `m_currentTheme` directly from MainWindow). MainWindow
  drives the freshness contract: every restyle path that
  changes `m_currentTheme` must call
  `m_claudeStatusBarController->applyTheme(m_currentTheme)` immediately
  after — the existing centralised restyle helper already
  fans out to all chrome on theme change, so this is one
  added line.
- **Provider lambdas hold a captured `[this]`** referring to
  MainWindow. When MainWindow is destroyed, the controller is
  destroyed first (QObject parent ordering: it's parented to
  `this` = MainWindow). No dangling-this risk.
