# ANTS-1145 — Extract DiffViewerDialog from mainwindow.cpp

- **Status:** Implementing (target 0.7.73)
- **Source:** indie-review-2026-05-01 (L6 LoC1) + cold-eyes review 2026-05-02
- **Lanes:** mainwindow, diffviewer (new)
- **Kind:** refactor (no behavioural change)

## Problem

`mainwindow.cpp` is 6636 lines. `MainWindow::showDiffViewer`
(currently lines 5747-6176, ~430 LoC) is the largest single
self-contained block in the file: it constructs a Qt dialog, runs
five async git probes, renders coloured HTML, owns a
`QFileSystemWatcher` + debounce timer, preserves scroll position
across live refreshes, and has only three references holding it
inside `MainWindow`:

- `m_currentTheme` (read once at open) — already snapshotted into a
  `themeName` local at the top of the function
- `m_claudeReviewBtn` (disabled on open, re-enabled on dialog
  destroy) — the call site already knows the button (it's what
  invoked the slot)
- `refreshReviewButton()` (called in the destroyed lambda) — the
  call site can wire this itself

Splitting this carve-out into its own translation unit (a) shrinks
`mainwindow.cpp` toward the eventual sub-2000-LoC target,
(b) gives the diff-viewer dialog a clear test surface (one .cpp
to grep, not one function inside a 6636-line file), and (c) is
the simplest of the three L6 LoC carve-outs (1146 / 1147 are
larger and tangled with theming + status-bar plumbing
respectively).

## Fix

New translation unit `src/diffviewer.{cpp,h}` exposing one free
function in a `diffviewer` namespace:

```cpp
namespace diffviewer {

// Build and show the Review Changes dialog for `cwd`. Returns the
// dialog (parented to `parent`, WA_DeleteOnClose) so the caller
// can connect to its destroyed() signal for post-close work.
QDialog *show(QWidget *parent,
              const QString &cwd,
              const QString &themeName);

}  // namespace diffviewer
```

### Surface-design choice (justified)

The free function takes only the data the dialog renders (`parent`,
`cwd`, `themeName`) and returns the `QDialog*` so the caller wires
its own `destroyed`-signal lambda for post-close work. Considered
and rejected: a richer 5-arg `void show(parent, cwd, themeName,
QPushButton *gateBtn, std::function<void()> onClosed)` that hides
the disable/re-enable inside the function. Reasoning:

- The dialog's responsibility is rendering diff state, not
  managing an arbitrary external button. Coupling
  `diffviewer::show` to `QPushButton*` makes it harder to call
  from a future command-palette entry or remote-control verb
  that doesn't have a "Review Changes" button to gate.
- The "boilerplate" at the call site (disable button → QPointer
  guard → connect destroyed → re-enable + refreshReviewButton)
  is five lines of idiomatic Qt; it's standard, not novel.
- Rule of Three: there is one call site today. Designing for a
  hypothetical second call site that needs button-gating
  semantics is the kind of premature DRY global rule §3 warns
  against.

`MainWindow::showDiffViewer()` shrinks to ~25 LoC:

1. Status message ("Review Changes: opening…")
2. Focused-terminal + cwd lookup (early return on missing)
3. Disable `m_claudeReviewBtn`, snapshot it into a `QPointer`
4. Call `diffviewer::show(this, cwd, m_currentTheme)`
5. Connect dialog `destroyed` → re-enable button + call
   `refreshReviewButton()`

### Behavioural parity (explicit)

Every probe, the `ProbeState` shape (`pending = 5`, `branches`,
`crossUnpushed`), the `lastHtml` byte-identity cache, the
`QFileSystemWatcher` paths, the debounce interval (300 ms), the
scroll-preservation two-layer guard, the Copy-Diff payload format,
and the colour-coded HTML rendering all move byte-for-byte. No
CSS / layout / wording changes — every user-visible string is
preserved (locked by INV-2c).

Two semantics worth naming explicitly so a future reader doesn't
"fix" them:

- **Theme is captured at open** by value into the lambdas (same as
  today). A theme switch while the dialog is live continues
  rendering in the dialog's open-time theme until close. Passing
  the theme by `const QString&` snapshot rather than a
  `MainWindow*` makes this invariant explicit.
- **QProcess parent moves from MainWindow to the dialog.** The
  original `new QProcess(this)` parented every probe to MainWindow
  — they outlived the dialog. Post-extraction the natural parent
  is the dialog, so an in-flight probe at dialog-close time gets
  killed instead of completing into nothing. Strict improvement
  (no zombie `git` processes from a fast user-close), worth a
  note in the post-merge journal.

### Include hygiene

- `mainwindow.cpp`: includes that lose their referent
  (`<QFileSystemWatcher>`, `<QTextEdit>`, `<QScrollBar>`,
  `<QProcess>`, `<QTimer>`, `clipboardguard.h` if no other use,
  `<QFileInfo>` if no other use) get removed. Verified by
  successful compile + grep.
- `diffviewer.h`: forward-declares `QWidget`, `QDialog`; full
  include for `QString`. No transitive Qt explosions on consumers.
- `diffviewer.cpp`: full includes for everything the dialog body
  uses (`QFileSystemWatcher`, `QTextEdit`, `QScrollBar`,
  `QProcess`, `QPushButton`, `QTimer`, `QVBoxLayout`,
  `QHBoxLayout`, `QLabel`, `QFileInfo`, `QPointer`, `QObject`,
  `QFont`, `clipboardguard.h`, `themes.h`).

## Acceptance (INVs)

Tests live in `tests/features/diffviewer_extraction/`. Source-grep
only — the dialog itself stays exercised by the existing four
`review_changes_*` tests after they're updated per the per-test
table in INV-5.

CMake wiring: `LABELS "features;fast"`, source-grep only (no
Qt6::Core link, no diffviewer.cpp link, just three compile-time
`SRC_*_PATH` defines).

- **INV-1** `src/diffviewer.h` exists and declares the free
  function with exact signature
  `QDialog *show(QWidget *parent, const QString &cwd, const
  QString &themeName);` (asserted by substring grep — locks
  parameter order/types so a "minor cleanup" PR can't silently
  reorder them).
- **INV-2a** `src/diffviewer.cpp` exists and contains the
  structural markers: `reviewChangesDialog` objectName,
  `struct ProbeState`, `pending = 5`, `for-each-ref` git probe,
  `QFileSystemWatcher`, `runProbes` lambda name, `lastHtml`
  shared cache.
- **INV-2b** `src/diffviewer.cpp` contains the user-visible
  strings byte-identical to the original — locks against
  drive-by reformatting:
  - Window title `"Review Changes"`
  - Button labels `"Refresh"`, `"Close"`, `"Copy Diff"`
  - Live-status texts `"● refreshing…"` and
    `"● live — auto-refresh on git changes"`
  - Section headers `"Status"`, `"Unpushed commits (current
    branch)"`, `"Branches"`, `"Unpushed across all branches"`,
    `"Diff"`
  - Empty-state line `"No status, diff, or unpushed commits to
    report."`
- **INV-2c** Sub-objects retain object names so the existing
  feature-coverage QA helpers can find them: `reviewLiveStatus`,
  `reviewRefreshBtn`.
- **INV-3a** `MainWindow::showDiffViewer` post-extraction body is
  ≤ 50 physical lines counted between the opening `{` and
  closing `}` exclusive of (a) blank lines and (b) lines whose
  first non-whitespace character is `//`. The cap is generous
  guard against accidental re-inlining, not a precise budget.
- **INV-3b** `MainWindow::showDiffViewer` body contains the
  substring `diffviewer::show(`. (Pre-extraction code fails
  this; pre-fix can also fail INV-3a if the implementer never
  shrinks the function.)
- **INV-4** `mainwindow.cpp` no longer contains `struct
  ProbeState`, `for-each-ref`, `runProbes`, or
  `reviewChangesDialog` — all moved to `diffviewer.cpp`.
