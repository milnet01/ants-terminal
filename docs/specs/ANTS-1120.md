# ANTS-1120 — Companion-instrumentation gate

- **Status:** Draft (awaiting user sign-off)
- **Owner:** Project lead
- **New bullet introduced by [ADR-0002](../decisions/0002-cold-eyes-companion-cleanup.md)** — gates ANTS-1110 / 1111 / 1112 / 1113 / 1114 / 1116 / 1117.
- **Source:** cold-eyes review 2026-04-30

## Why this bullet exists

Every Claude-Code companion bullet justifies itself with a
"saves N tokens" argument. None of those numbers are measured.
ANTS-1116 promises 10× composite reduction; ANTS-1117 promises
25× on `audit-run`; ANTS-1110 promises a global disposition shift.
Without measurement, we'll over-invest in low-leverage paths and
under-invest in high-leverage ones.

This bullet **does not ship a new feature**. It ships a one-time
measurement run plus the discipline to gate further companion work
on the resulting numbers.

## Scope

A scripted measurement of **one Claude Code task replayed at
least three times in each configuration** — N≥3 is mandatory
because Claude responses are non-deterministic; one run
produces a noisy point estimate, three runs let us bound the
variance.

The two configurations:

- **Configuration A (baseline):** the user's current workflow —
  Claude invokes the audit tooling directly, reads the SARIF
  output, summarises in-context, allocates a ROADMAP ID by
  reading `.roadmap-counter` + writing it back, and so on for
  every instrumented surface.
- **Configuration B (instrumented):** the same task shape, but
  with stub responses for `audit-run`, `roadmap-query`, and
  `id-allocate`. The stubs simulate what ANTS-1116 + ANTS-1117
  would produce.

**Stub sourcing.** Stubs are not invented from imagination —
they are **captured from one real run of the production
tooling** before the measurement begins:

- `audit-run` stub = the actual JSON shape `auditengine` (per
  ANTS-1119) emits when the audit pipeline finishes against
  this repo's tree.
- `roadmap-query` stub = the actual JSON shape
  `RoadmapDialog::parseBullets` produces against the current
  ROADMAP.md.
- `id-allocate` stub = a fixed pre-allocated ID + counter
  bump.

Token counts come from the **Anthropic API `usage` field** on
each response — not estimated, not derived from `wc -c` on the
prompt. The measurement script captures every API request /
response pair into a JSON Lines log.

The deliverable is a single artifact:
**`docs/journal/ANTS-1120-measurement.md`** with:

- Per-prompt, per-run input + output token counts for both
  configurations.
- Composite token totals per run.
- Mean + standard deviation across the 3 runs per
  configuration.
- Measured ratio (`baseline_mean / instrumented_mean`) and a
  **conservative ratio** (`baseline_mean - 1σ` over
  `instrumented_mean + 1σ`).
- A keep / iterate / drop / inconclusive verdict per
  downstream bullet — **threshold values are decided by the
  user when the measurement lands**, not pre-set in this
  spec. (Cold-eyes finding 19-3: arbitrary thresholds in the
  spec would be ungrounded.)

## Out of scope

- **Implementing the production helpers.** This bullet uses
  *stubs*. The real implementations land under ANTS-1116 /
  ANTS-1117 once the verdict is recorded.
- **Multi-session averaging.** One real session is enough to
  rule out 10× claims that aren't true.
- **Cost-in-currency math.** Token-count ratio is the
  measurement. Currency conversion (cited as the user's ZAR
  motivation) is downstream of token count and not the
  variable we control.

## Acceptance criteria (INVs)

- **INV-1** The recorded task exercises at least three of the
  instrumented surfaces (audit, roadmap query, ID allocation
  — minimum, more if other surfaces are added before the
  measurement runs).
- **INV-2** Each configuration runs **N ≥ 3 times**; the
  prompt sequence is byte-identical across runs (the
  measurement script replays the same input, only the tool
  outputs differ between A and B).
- **INV-3** The journal artifact contains per-prompt
  per-run token counts (input + output + cache_read + cache_creation)
  extracted from the API `usage` fields, not estimated.
- **INV-4** The journal artifact reports both the
  point-estimate ratio and a conservative ratio (1σ
  pessimistic on baseline, 1σ optimistic on instrumented).
- **INV-5** The journal artifact records an explicit
  per-downstream-bullet verdict
  (keep / iterate / drop / inconclusive) chosen by the user
  *after* reviewing the measurement, with a one-line
  justification. The user picks the keep/iterate/drop
  thresholds at this point.
- **INV-6** Verdict is reflected back into ROADMAP.md:
  - **Keep** → bullet stays 📋 with no body change.
  - **Iterate** → bullet body trimmed; updated `Source:` line
    cites the measurement (`Source:
    measurement-ANTS-1120-2026-MM-DD`).
  - **Drop** → bullet retired with a `**Retired:** superseded
    by measurement-ANTS-1120` note inline; status emoji
    removed (the bullet survives as a documentation
    breadcrumb so future readers see why the work didn't
    happen — per `roadmap-format.md` § 3.5.1 (stable-ID
    immutability), only bodies change).
  - **Inconclusive** → bullet body annotated "needs deeper
    measurement"; user decides whether to re-run with bigger
    N or drop entirely.

## Test plan

This bullet doesn't ship code, so there's no feature-test
file. The lock is the journal artifact + the visible ROADMAP
edits. Acceptance is the user reading the journal and
agreeing the measurement is honest.

## Deliverables

1. `docs/specs/ANTS-1120.md` (this file).
2. `scripts/measure-companion-tokens.sh` — drives the
   side-by-side run.
3. `docs/journal/ANTS-1120-measurement.md` — measurement
   record + per-bullet verdict.
4. ROADMAP.md edits per the verdict.

## Open questions for sign-off

1. The N≥3 replays per configuration covers the determinism
   cold-eyes finding. If the variance is still high, do we
   bump to N≥5 or accept the wider error bars and flag the
   bullet as "needs deeper measurement"?

## Cold-eyes review pass 1 (2026-04-30)

Findings folded back in:

- 19-1 Resolved: N≥3 replays per configuration; mean + σ
  reported; conservative ratio (1σ pessimistic on baseline,
  1σ optimistic on instrumented) added to deliverable.
- 19-2 Resolved: stub-recording method specified — captured
  from real production tooling output before measurement
  begins, not invented.
- 19-3 Resolved: 3×/5× thresholds removed from spec; user
  picks thresholds at measurement-review time.
- 19-4 Resolved: INV-6 lays out the four-way disposition
  (keep/iterate/drop/inconclusive) explicitly, with the
  "drop" case using a `**Retired:**` body annotation rather
  than the misleading 💭 status emoji.
- ID renumbering: was ANTS-1119 in the original draft; renamed
  to ANTS-1120 because ANTS-1119 now covers the audit-engine
  extraction prep work (see ADR-0002 decision 4).
