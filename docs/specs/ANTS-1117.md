# ANTS-1117 — Claude Code → running-Ants GUI IPC (v1)

- **Status:** Draft (awaiting user sign-off)
- **Owner:** Project lead
- **Promoted ahead of ANTS-1116** per [ADR-0002](../decisions/0002-cold-eyes-companion-cleanup.md)
- **Source:** user-2026-04-30 + cold-eyes review 2026-04-30

## Why this ships first

Of the two integration bullets, this one delivers the **highest-value
verbs through the smallest diff**:

- The IPC transport (`src/remotecontrol.{h,cpp}`) already exists,
  ships via a Unix-domain socket in `$XDG_RUNTIME_DIR`, has its
  trust model gated behind the existing `remote_control_enabled`
  config key (default off, 0700 perms — verified by ANTS-1018 in
  0.7.12).
- The verbs land on top of code paths already exercised by the GUI
  user (audit dialog, roadmap dialog, tab list). No duplicated
  parsing, no second copy of audit logic.
- One real token-saving channel (`audit-run` → triaged JSON)
  replaces the slowest current flow (Claude reads SARIF + summarises).

## Scope (v1)

Two new verbs added to `src/remotecontrol.cpp`. **No new transport,
no new authentication, no new config keys.**

| Verb | Calls into | Returns |
|------|-----------|---------|
| `roadmap-query` | `RoadmapDialog::extractToc` + a new `RoadmapDialog::parseBullets(markdown)` to be extracted as part of this bullet's implementation (only `extractToc` exists in `roadmapdialog.{h,cpp}` today) | `{"bullets": [{"id": "ANTS-NNNN", "status": "✅"\|"🚧"\|"📋"\|"💭", "headline": "...", "kind": "...", "lanes": [...]}, ...]}` |
| `tab-list` | New `MainWindow::tabsAsJson()` helper consulting `m_tabs` / shell-PID detection / claude-detection | `{"tabs": [{"index": N, "title": "...", "cwd": "...", "shell_pid": N, "claude_running": bool, "color": "..."}]}` |

Both verbs are read-only — they don't mutate any state. Out of v1:
`audit-run` (touches the long-running audit pipeline; bigger surface;
ships in v2 once measurement validates the lift), `tab-close`,
`tab-rename`, `terminal-write`, `theme-apply`, `screenshot-capture`.

## Wire path

Claude Code calls the existing remote-control entry point through the
`Bash` tool. Confirmed by reading `src/remotecontrol.cpp`:

- The dispatch is a flat if-chain on `cmd == QLatin1String("ls" | "send-text" | "new-tab" | "select-window" | "set-title" | "get-text" | "launch")` — see `remotecontrol.cpp` ~line 162. Two new branches (`"roadmap-query"` and `"tab-list"`) join this chain.
- Client invocation goes through the existing static
  `RemoteControl::runClient(command, args, socketPath)` —
  declared in `remotecontrol.h:66`. No new client function is
  added; Claude calls the same `ants-terminal --remote-control`
  invocation surface that today drives `ls` / `set-title` /
  `get-text`.
- The socket path is read from the `ANTS_REMOTE_SOCKET` env var
  (set by the running instance; default falls back to
  `defaultSocketPath()`). The env var is already exposed —
  verified at `remotecontrol.cpp:54`.
- Existing trust model: `lstat`-checked `S_ISSOCK` + UID match
  (per `remotecontrol.cpp:31-36`). New verbs inherit it.

## Error-response shape (unified)

All verb responses follow one of two shapes:

```json
{ "ok": true,  "<verb-fields>": ... }              // success
{ "ok": false, "error": "...", "code": "..." }    // server-side refusal / handler error
```

The success shape is **flat** — verb-specific fields sit at the top
level alongside `ok`. Concrete examples (the codebase's pre-existing
`ls` / `get-text` / `tab-list` / `roadmap-query` precedent):

```json
{ "ok": true, "tabs":    [ { ... } ] }    // tab-list
{ "ok": true, "bullets": [ { ... } ], "path": "...", "count": N }   // roadmap-query
```

A nested `{ "ok": true, "data": { ... } }` envelope was considered
during the v1 draft but rejected at fold-in time (ANTS-1123
indie-review F9): the codebase's three pre-existing read-only verbs
(`ls`, `get-text`, plus the future `tab-list`) all use the flat shape,
so a nested-on-`data` exception for `roadmap-query` would be the
outlier. Future mutating verbs (e.g. `audit-run` in v2) follow the
flat-on-success convention by default.

Where:
- `code` is a stable string matching `^[a-z][a-z_-]*$` so callers can
  switch on it (e.g. `"feature_disabled"`, `"no_roadmap_loaded"`,
  `"parse_error"`). Human-readable `error` messages are not stable.
- Client-side connection errors (socket-missing, EPIPE, etc.) are
  generated by the client transport, not the verb handler — those
  return whatever the existing remote-control client returns and are
  out of this spec's contract.

## Out of scope

- **Authentication.** UID-scope + 0700 perms is enough for v1 (matches
  the existing transport's model).
- **Streaming responses.** Both verbs return a single JSON document.
- **Mutating verbs.** None ship in v1. v2 adds `audit-run` because it
  has the highest token-saving leverage; later verbs follow per
  ANTS-1119 measurement.
- **Network exposure.** The socket stays Unix-domain-only.

## Acceptance criteria (INVs)

- **INV-1** `roadmap-query` returns one entry per top-level
  status-emoji-prefixed bullet in the loaded ROADMAP.md, in
  document order. ("Top-level" means `^- ` or `^* ` at column 0;
  continuation indent is part of the prior bullet.)
- **INV-2** Each `roadmap-query` entry contains a stable `id`
  field matching `ANTS-\d+` per
  [`docs/standards/roadmap-format.md`](../standards/roadmap-format.md)
  § 3.5.1 (stable IDs — `[PROJ-NNNN]`).
- **INV-3** `roadmap-query` recognises and surfaces the four legend
  emojis (`✅` `📋` `🚧` `💭`) defined in roadmap-format.md § 3.3
  (status emojis);
  plain-narration bullets without an emoji are omitted from the
  response (they don't have IDs and so are out of contract).
- **INV-4** `roadmap-query` is read-only — running it twice in
  succession produces byte-identical output (idempotent; no
  mutation of file, dialog state, or cache).
- **INV-5** `tab-list` returns one entry per `m_tabs->count()`,
  zero-indexed `index` field, sorted ascending.
- **INV-6** `tab-list`'s `claude_running` field is `true` iff the
  same per-tab Claude detection the status bar already consults
  reports active. Concretely: `ClaudeTabTracker::shellState(pid)
  .state != ClaudeState::NotRunning` (per
  `src/claudetabtracker.h:33-47`). The verb handler reads through
  the existing `MainWindow::m_claudeTabTracker` member; no new
  detection logic.
- **INV-7** **Server-side refusal when `remote_control_enabled`
  is false but the binary is otherwise running:** the listener
  isn't bound (config gates the socket creation), so client-side
  invocations return the existing transport's "connection
  refused" / "no socket" error — no `roadmap-query` /
  `tab-list` handler ever runs. (This INV documents the contract
  but the test fixture is a server-side observability check —
  i.e. that no listener thread is spawned when the config bit
  is off.)
- **INV-8** When the socket exists but the running instance has
  no loaded `m_roadmapPath`, `roadmap-query` returns the
  unified-shape error `{"ok": false, "error": "no roadmap
  loaded for the current tab", "code": "no_roadmap_loaded"}`
  with HTTP-style 200 success on the transport (the verb ran
  fine; the failure is *semantic*, not transport).
- **INV-9** Both verbs round-trip valid UTF-8 unchanged: no
  Unicode normalisation (NFC/NFD), no escape-sequence rewriting,
  no BOM injection. Emoji bytes ≥ U+10000 survive byte-identical.
- **INV-10** Verb-handler invocation rate-limit:
  `roadmap-query` parses the file fresh on each call but the
  parsed model is cached for `≤ 100 ms` so a misbehaving caller
  can't burn CPU at 1000-req/s. Cache invalidates on
  `QFileSystemWatcher::fileChanged` for `m_roadmapPath`.
- **INV-11** Both verbs respond inside `≤ 250 ms` on a
  3000-line ROADMAP.md / 20-tab MainWindow steady state — a
  perf invariant the roadmap viewer's existing benchmark
  hooks can verify.

## Test plan

`tests/features/remote_control_roadmap_query/` and
`tests/features/remote_control_tab_list/`. Each follows the pattern of
existing `tests/features/remote_control_*` cases:

- Source-grep INV: verb registration in `remotecontrol.cpp`, calls
  into the named helpers.
- Behavioural INV: link `remotecontrol.cpp` + `roadmapdialog.cpp`,
  invoke the verb handler against synthetic input, assert the
  JSON output schema.
- Trust INV: assert the verb refuses to run when
  `remote_control_enabled` is false.

- 17-7 Resolved: § Error-response shape rewritten to match the
  codebase precedent (flat `{ok, <fields>}` on success). The
  earlier draft mandated a nested `{ok, data}` envelope; that

  the flat shape is locked by four existing verb tests and by

  through more surfaces than changing the doc.
