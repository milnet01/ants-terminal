# ANTS-1147 — Extract themedstylesheet from mainwindow.cpp

- **Status:** Implementing (target 0.7.74, bundled with ANTS-1146)
- **Source:** indie-review-2026-05-01 (L6 LoC3) + brainstorm 2026-05-02
- **Lanes:** mainwindow, themedstylesheet (new)
- **Kind:** refactor (no behavioural change) + enables one perf
  follow-up (the `updateStatusBar` cache-and-compare optimization
  the roadmap entry described).

## Problem

`MainWindow::applyTheme(const QString &name)` (lines 2798-3140 of
the post-1146 `mainwindow.cpp`) is the central restyle path. Its
body mixes two concerns:

1. **String-building.** A ~178-LoC `QString` template assembled via
   `QString::arg()` from the active theme's six colour roles, then
   pushed to `qApp->setStyleSheet`. Plus six smaller per-widget
   templates (menubar, branch chip, git separator, status message,
   status process, plus a near-identical chip template inside
   `refreshRepoVisibility` at line 4911).
2. **Side effects.** `qApp->setStyleSheet`, the dialog re-polish
   loop, `m_titleBar->setThemeColors`, three `setBackgroundFill`
   calls, three `setPalette` calls, and three `update()` calls.

The string-building half is a pure function of `(Theme, …flags)`
— ideal carve-out target. The side effects belong to MainWindow
and stay there. Splitting (a) shrinks `applyTheme` toward a
readable method size, (b) makes the QSS template independently
testable (no widget instantiation needed), (c) **enables the
roadmap-stated perf optimization**: `updateStatusBar`
(`mainwindow.cpp:4121`) currently re-applies the branch-chip
stylesheet on every 2 s tick (line 4180) even when neither the
theme nor the primary-branch flag has changed. With the QSS as a
pure function, MainWindow can cache the most-recent
`(theme, primary)` pair and the resulting stylesheet, only calling
`setStyleSheet` when one of those changes.

(d) `refreshRepoVisibility` and `applyTheme` both inline a
near-identical "chip" QSS template. Sharing the helper closes a
silent drift hazard — change one, miss the other.

## Fix

New translation unit `src/themedstylesheet.{cpp,h}` exposing
pure functions in a `themedstylesheet` namespace:

```cpp
namespace themedstylesheet {

// The big app-level cascade applied via qApp->setStyleSheet.
// Covers QMainWindow / QMenu* / QStatusBar / QTabBar / dialog
// chrome / QPushButton / QLineEdit / QCheckBox / etc. Includes
// the data-URI SVG tab-close glyph.
QString buildAppStylesheet(const Theme &theme);

// Per-widget templates — each takes only the data the QSS needs.
QString buildMenuBarStylesheet(const Theme &theme);
QString buildStatusMessageStylesheet(const Theme &theme);
QString buildStatusProcessStylesheet(const Theme &theme);
QString buildGitSeparatorStylesheet(const Theme &theme);

// Chip stylesheet shared by m_statusGitBranch and
// m_repoVisibilityLabel — the only delta is the left margin
// (4 px on the branch chip, 0 on the repo-visibility chip), so
// the helper takes it as a parameter. Foreground colour comes
// from the caller because the branch chip picks green/amber by
// primary-branch flag and the visibility chip picks green/red
// by Public/Private — different policies, same QSS shape.
QString buildChipStylesheet(const Theme &theme,
                            const QColor &fgColor,
                            int leftMarginPx);

}  // namespace themedstylesheet
```

### Surface-design choice (justified)

Three options were enumerated:

- **(A)** Pure functions in a namespace. Stateless. Matches
  ANTS-1145's free-function pattern. Easy to test: each is
  `(input) → QString`, no widget needed.
- **(B)** Helper class instance owned by MainWindow. No state to
  own — rejected (YAGNI).
- **(C)** Inline static helpers (file-local lambdas). Doesn't
  carve anything out — keeps the QSS in mainwindow.cpp.

**Decision: (A).** The helpers are pure functions with no
lifetime, no state, no I/O. Namespace gives them a logical home
(`themedstylesheet::*`) without forcing a class.

### Behavioural parity (explicit)

Every QSS template moves byte-for-byte from `mainwindow.cpp` to
`themedstylesheet.cpp`. The only mechanical changes are:

- The `QString::arg(...)` argument-pack moves from inline at the
  call site into the helper body.
- `theme.ansi[1].name()` (and the two `QStringLiteral("%23") +`
  splices for the SVG-stroke pre-encoding inside
  `buildAppStylesheet`) move into the helper body. **The
  explanatory comment block at mainwindow.cpp:2996-3008 (about
  Qt's CSS parser truncating at `#` and the
  `QString::arg()` placeholder-numbering collision) MUST migrate
  verbatim alongside the splices.** Without the comment a future
  reader sees `QStringLiteral("%23") + name.mid(1)` and assumes
  it's dead code.
- Per-call-site arguments (`primaryBranch` flag for the chip,
  `fgColor` for the chip, `leftMarginPx` for the chip) become
  explicit parameters.
- All three pre-1147 chip sites (`applyTheme:3118`,
  `updateStatusBar:4180`, `refreshRepoVisibility:4911`) become
  callers of `themedstylesheet::buildChipStylesheet`.

Two semantics named explicitly:

- **`buildChipStylesheet` is shared** across **three** call sites:
  `applyTheme` at the previous line 3118 (initial chip styling
  on theme change), `updateStatusBar` at the previous line 4180
  (per-tick chip restyle for branch-color changes), and
  `refreshRepoVisibility` at the previous line 4911 (visibility
  badge styling on cache hit). Pre-1147 these were three
  near-identical copy-pastes that diverged only in the left
  margin (4 px on the two branch-chip sites, 0 on the visibility
  badge); post-1147 they're three callers of one helper with an
  int parameter.
- **`updateStatusBar` cache-and-compare optimization is part of
  the deliverable.** Four new private members on MainWindow:
  `bool m_lastBranchChipValid = false` (validity flag — false
  on first call so the first tick always applies), `QString
  m_lastBranchChipQss`, `bool m_lastBranchChipPrimary`,
  `QString m_lastBranchChipTheme`. Tick logic:
  ```cpp
  const QString newQss = themedstylesheet::buildChipStylesheet(
      chipTheme, chipCol, /*leftMarginPx=*/4);
  if (!m_lastBranchChipValid ||
      newQss != m_lastBranchChipQss) {
      m_statusGitBranch->setStyleSheet(newQss);
      m_lastBranchChipQss = newQss;
      m_lastBranchChipPrimary = chipPrimary;
      m_lastBranchChipTheme = m_currentTheme;
      m_lastBranchChipValid = true;
  }
  ```
  The QSS string itself encodes the theme + primary-flag + margin
  triple (different inputs produce different strings), so the
  string-comparison is the canonical guard. The `m_lastBranchChipPrimary`
  / `m_lastBranchChipTheme` members are kept for diagnostics + so
  a future bullet can extend the cache to other inputs without
  recomputing the QSS to inspect them. Same shape applies to
  `refreshRepoVisibility` but is left for a future bullet — out
  of scope here per Karpathy rule 11 (stay in your lane).

### Include hygiene

- `mainwindow.cpp`: no includes lose a referent (the QSS body
  used `QString` / `QStringLiteral` / `QColor` which all stay).
  Adds `#include "themedstylesheet.h"`.
- `mainwindow.h`: adds three private members
  (`m_lastBranchChipQss`, `m_lastBranchChipPrimary`,
  `m_lastBranchChipTheme`).
- `themedstylesheet.h`: forward-declares `QColor`. Full include
  for `<QString>`. Includes `themes.h` (member-typed `const Theme &`).
- `themedstylesheet.cpp`: full includes for `<QString>`,
  `<QStringLiteral>` (via `<QString>`), `themes.h`. No Qt widget
  headers — these helpers don't touch QWidget.

## Acceptance (INVs)

Tests live in `tests/features/themedstylesheet_extraction/`.
Source-grep + unit-level (the helpers are pure, so a small
non-GUI test can call them directly and assert string contents).

- **INV-1** `src/themedstylesheet.h` exists and declares
  `namespace themedstylesheet` with the six public function
  signatures byte-for-byte (each substring grep'd):
  - `QString buildAppStylesheet(const Theme &theme);`
  - `QString buildMenuBarStylesheet(const Theme &theme);`
  - `QString buildStatusMessageStylesheet(const Theme &theme);`
  - `QString buildStatusProcessStylesheet(const Theme &theme);`
  - `QString buildGitSeparatorStylesheet(const Theme &theme);`
  - `QString buildChipStylesheet(const Theme &theme, const QColor &fgColor, int leftMarginPx);`
- **INV-2** `src/themedstylesheet.cpp` contains the migrated QSS
  selectors byte-for-byte. Locks against drive-by reformatting.
  Note: `QProgressBar::chunk` is intentionally NOT in this list —
  the same selector also lives in `claudestatuswidgets.cpp` (the
  context-bar chunk colour, ANTS-1146). A non-unique marker
  would still pass INV-2's "must appear here" check but would
  weaken any future "must NOT appear elsewhere" check, so we
  exclude it.
  - App-level selectors: `QMainWindow {`, `QMenuBar::item:hover`,
    `QMenuBar::item:selected`, `QTabBar::tab:selected`,
    `QTabBar::close-button {`, `QPushButton:hover:enabled`,

- **INV-4** `mainwindow.cpp::applyTheme` body calls each of the

  - `themedstylesheet::buildMenuBarStylesheet(`

- **INV-6** `MainWindow::updateStatusBar` body contains the

  - `mainwindow.h` declares all four cache members:
    `m_lastBranchChipValid`, `m_lastBranchChipQss`,

- **INV-7** Two-sided LoC anchor:
  - `themedstylesheet.cpp` LoC ≥ 200.
  - `mainwindow.cpp` LoC strictly decreases by ≥ 200 vs. the

    templates and `.arg(...)` plumbing − ~15 LoC of helper calls
    left in place) puts the expected delta around 213, so a 150
    floor would let a partial extraction pass.

### INV-8 — Unit-level helper tests

Direct calls into `themedstylesheet::*` from the test, asserting
the returned `QString` for a fixed theme matches expected
substrings. The full QSS is too long to assert byte-for-byte, but
a handful of structural anchors guards against silent corruption:

- `buildAppStylesheet(Themes::byName("dark"))` returns a string
  that contains `QMainWindow { background-color:` AND
  `QPushButton:hover:enabled` AND `data:image/svg+xml;utf8`.
- `buildChipStylesheet(theme, QColor("#00ff00"), 4)` returns a
  string containing `color: #00ff00`, `margin: 2px 6px 2px 4px`,
  `border-radius: 3px`, `font-weight: 600`.
- `buildChipStylesheet(theme, QColor("#ff0000"), 0)` returns a
  string containing `margin: 2px 6px 2px 0` (zero left margin —
  the repo-visibility variant).
