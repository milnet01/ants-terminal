# ANTS-1106 — Mandatory `Kind:` field + viewer faceted categorisation

- **Status:** Implementing (pre-0.7.64)
- **Source:** user-2026-04-30
- **Lanes:** docs, RoadmapDialog

## What v1 ships

The `Kind:` field backfill across every existing ROADMAP bullet
landed alongside the original 1106 commit (status flips +
backfill). The standards doc (`docs/standards/roadmap-format.md`
§ 3.5.3) was updated to mark `Kind:` as required. What's left is
the viewer half:

A **secondary faceted filter** in `RoadmapDialog`, sitting
underneath the existing ANTS-1100 status-filter checkbox row,
that lets the user pick **one or more `Kind:` values** to focus
on. The implementation reuses the existing renderer pipeline —
no new walker, no second copy of the bullet-detection logic.

## Filter semantics

- **Empty filter (default) = show all.** Every bullet that
  passed the status + search filters renders unchanged.
- **Non-empty filter = OR-include.** Only bullets whose `Kind:`
  line value matches one of the active kinds render. Bullets
  with **no `Kind:` line** are excluded when the filter is
  non-empty (the spec says Kind is required; absence of the
  field is a backfill miss, and the user filtering by Kind
  doesn't want unclassified bullets in the results).

## UI shape

A new `QHBoxLayout` row added to the dialog's vertical layout
*after* the status-checkbox row, *before* the splitter body. The
row contains:

- A small label `Kind:` on the left.
- Ten `QCheckBox` widgets, one per Kind value the standards doc
  enumerates (`implement`, `fix`, `audit-fix`, `review-fix`,
  `doc`, `doc-fix`, `refactor`, `test`, `chore`, `release`),
  plus the de-facto values present in the current ROADMAP
  (`research`, `ux`). Order matches the standards doc.
- Each checkbox label uses an emoji prefix as a colour cue
  (e.g. `✨ implement`, `🐛 fix`, `🔍 audit-fix`, `📚 doc`).
  All start unchecked.
- A trailing stretch so the row left-aligns.

User toggles: each checkbox's `toggled` signal updates the
`m_kindFilter` `QSet<QString>` member and triggers the existing
`refresh()` re-render path.

## Implementation summary

- `RoadmapDialog::renderHtml` gains a defaulted 7th parameter
  `const QSet<QString> &kindFilter = {}`. Existing callers that
  pass 6 args continue to work.
- Inside the existing bullet-emission walk in `renderHtml`,
  after the current `keepStatus` / `keepSearch` checks, peek
  ahead to assemble the bullet's full body (continuation lines
  indented by two spaces) and extract the `Kind:` line via the
  same regex `parseBullets` already uses. Add a
  `keepKind = kindFilter.isEmpty() || (!thisKind.isEmpty() &&
  kindFilter.contains(thisKind))` check; combine into the
  existing `keep` boolean.
- New private member `QSet<QString> m_kindFilter`, populated on
  every checkbox toggle.
- Each checkbox is a `QPointer<QCheckBox>` member named
  `m_kindFilter<KindName>` for symmetry with the existing
  `m_filterDone` / `m_filterPlanned` etc. members.

## Out of scope

- Per-Kind colour cues beyond the emoji prefix. The spec calls
  for "audit-fix visually pops vs implement" — emoji prefix
  delivers that without a stylesheet rule per Kind. A future
  pass can add accent borders or theming if needed.
- "All Kinds" / "None" master toggle. Two-click to clear all
  is fine for v1; a master toggle is one of those features that
  reads as "needed" until users actually try it and find that
  unchecking-then-checking-the-one-you-want is the natural
  flow.
- Persisting the active Kind filter across sessions. The
  filter resets on each dialog open (matches the existing
  status-checkbox behaviour — they default to all-on every
  time).
- Updating the user-level template at
  `~/.claude/skills/app-workflow/templates/docs/standards/roadmap-format.md`.
  In-tree standards doc is the source of truth for this
  project; template alignment lands in a separate user-side
  edit when the user is ready.

## Acceptance (INVs)

Source-grep + behavioural drive of `renderHtml`. Tests live in
`tests/features/roadmap_kind_facets/`.

- **INV-1** `RoadmapDialog::renderHtml` accepts a defaulted 7th
  parameter `const QSet<QString> &kindFilter`. Asserted by
  source-grep on the header signature.
- **INV-2** When `kindFilter` is empty, `renderHtml` output
  contains every bullet that passes the status filter
  (existing behaviour — no regression). Behavioural drive.
- **INV-3** When `kindFilter` contains `"implement"`, only
  bullets whose body has `Kind: implement` are emitted.
  Behavioural drive on a synthetic 3-bullet markdown sample.
- **INV-4** When `kindFilter` contains multiple values, OR
  semantics apply (a bullet with `Kind: fix` is included
  when the filter contains `{"fix", "doc"}`). Behavioural
  drive.
- **INV-5** When `kindFilter` is non-empty and a bullet has
  no `Kind:` line at all, that bullet is excluded.
  Behavioural drive — the user filtering by Kind doesn't
  want unclassified bullets sneaking through.
- **INV-6** The dialog's vertical layout adds a Kind-filter
  row positioned below the status-checkbox row and above the
  splitter body. Asserted by source-grep on the construction
  code (the row's `QLabel("Kind:")` plus at least three of
  the checkbox object names — `roadmap-filter-kind-implement`,
  `roadmap-filter-kind-fix`, `roadmap-filter-kind-doc`).
- **INV-7** Each Kind checkbox has a stable `objectName`
  matching `roadmap-filter-kind-<kind>` so QSS targeting and
  test discovery have a hook. Asserted by source-grep.
