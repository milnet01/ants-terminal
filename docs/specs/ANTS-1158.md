> **Status:** spec — pre-implementation. Cold-eyes reviewed

> 5 HIGH / 3 MEDIUM / 4 LOW folded — see §13).
> **Source:** user request 2026-05-02 (verbatim)
>
> > "When Claude Code has a task list, please add a button to the
> > task bar that brings up a dialog that shows all the tasks. It
> > must show the task status (completed, running, outstanding)."
>
> **Lanes:** new `claudetasklist.{cpp,h}` + `claudetasklistdialog.{cpp,h}`;

> `MainWindow` (provider wiring + tab-switch retargeting).
> **Kind:** implement

## 1. Problem

Claude Code's TUI surfaces an in-progress task list (the running
plan from `TodoWrite` / `TaskCreate` / `TaskUpdate`) inline in the
chat pane. When the user is *not* looking at the Claude Code TUI —
e.g. they switched to a sibling tab, or the assistant collapsed
the list to a one-line summary — there's no way from inside Ants
Terminal to see the current plan or its progress.

## 2. Goal

A status-bar widget on the focused tab that, when a task list
exists, shows a count chip (e.g. `3/5` for in-progress + pending
out of total) and on click opens a dialog listing every task in
the current plan with its status:

- **completed** (✓)
- **in_progress** ("running" in user's wording — ◐)
- **pending** ("outstanding" in user's wording — ☐)

When no task list exists for the focused tab, the widget is
hidden.

## 3. Source of truth

Claude Code persists task state **only** in the active session
JSONL transcript at:

```
~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl
```

(`~/.claude/tasks/<session-uuid>/` exists but only holds `.lock`
and `.highwatermark` — no payloads. Verified 2026-05-02.)

### 3.1 Discriminating plan events from subagent dispatches

The Claude Code "Task" tool family overlaps two roles:

| Event shape | Role | Treat as plan event? |
|---|---|---|
| `name == "TaskCreate"`, `input` has `{subject, activeForm, ...}` | Add a row to the user-visible plan | **YES** |
| `name == "TaskUpdate"`, `input` has `{taskId, status, ...}` | Flip the status of a row | **YES** |
| `name == "TodoWrite"`, `input` has `{todos: [...]}` | Full snapshot of the plan (older Claude Code versions) | **YES — replaces list** |
| `name == "Task"` or `"Agent"`, `input` has `subagent_type` AND/OR `prompt` | Spawn a sub-agent (NOT a plan row) | **NO — filter out** |
| `name == "TaskList"` / `"TaskGet"` / `"TaskStop"` / `"TaskOutput"` | Read-only / control | **NO — ignore** |

Verified 2026-05-02 against the user's transcripts: TaskCreate
appears with the `{subject, description, activeForm}` shape —
that is the plan-add shape, distinct from the subagent-dispatch
shape (`prompt` + `subagent_type`). TodoWrite is absent from the
user's history but the parser supports it for forward-compat
with environments where it's used.

Additional filter: **events with `isSidechain == true` are
ignored**. Subagent turns inline their tool calls in the parent
transcript with that flag; including them would make a parent
chip flicker on subagent task progress that the user never
asked to see.

### 3.2 Replay algorithm

The plan list is reconstructed by walking events in two
modes:

**Mode A — TodoWrite-only (snapshot mode).** If the transcript
contains *any* TodoWrite event, the most recent one wins.
Walk **backwards from EOF** to find it (cheap since the
relevant event is usually near the tail), then return its
`input.todos` as the plan. Discards the entire TaskCreate /
TaskUpdate stream.

**Mode B — TaskCreate / TaskUpdate replay (incremental mode).**
Used when no TodoWrite was found. Walk forward from the start,
applying:

```
state = empty list
for each event in chronological order:
    if event.isSidechain: continue
    for c in event.message.content[]:
        if c.type != "tool_use": continue
        if name == "TaskCreate":
            id = scan_paired_tool_result(c.id)   // arrives async
            state.append({id, subject, status: "pending", ...})
        elif name == "TaskUpdate":
            t = state.find(input.taskId)
            if t: apply update fields onto t
return state
```

Notes:
- `TaskCreate`'s allocated ID arrives in the *paired*
  `tool_result` (matched via `tool_use_id`). Mirrors the
  bg-task launch/confirmation pattern at
  `claudebgtasks.cpp:222–243`.
- A `tool_result` for an unmatched `tool_use_id` is ignored
  (in-flight failure or concurrent agent).

### 3.3 Replay scope and bounds

- **One transcript** (the focused tab's session). No
  cross-session merge.
- **No 16 MiB tail cap on Mode A.** A naive tail-cap can drop
  the only TodoWrite snapshot on a long session — the snapshot
  is replace-not-append, so losing it loses *all* state. The
  EOF-walk-backward strategy bounds the work without the cap:
  read the file in 64 KiB blocks from the end, scan for
  `"name":"TodoWrite"` substring, parse only the matching line.
- **16 MiB tail cap on Mode B is acceptable.** TaskCreate /
  TaskUpdate are append-only, so the truncated head is older
  history; we only lose tasks created before that point. If
  the user has a 16+ MiB session with > N tasks per MiB,
  earlier-created tasks may show as missing — acceptable
  trade-off vs unbounded read time.
- **Threading: GUI thread only.** Static parser is `static`
  for testability (unit tests can call it without owning a
  tracker), but production callers always invoke from the GUI
  thread under the QFileSystemWatcher signal. Not safe to call
  off-thread on the same `QFile` handle (Qt I/O classes are
  not reentrant). Mirrors `claudebgtasks.cpp:137`.

## 4. Per-tab scoping

A user can run `claude` in multiple tabs concurrently. The
status-bar widget reflects the **focused** tab's task list. On
tab switch (`onTabChanged` already fires), retarget the tracker
to the new tab's `transcriptPath` (same pattern
`ClaudeBgTaskTracker::setTranscriptPath` uses).

The transcript path for a tab is already known —
`ClaudeTabTracker::ShellEntry::transcriptPath` carries it.
Look up via `m_tracker` on the focused shell PID.

## 5. Update cadence

- One `QFileSystemWatcher` on the active transcript path. On
  `fileChanged` re-parse and emit `tasksChanged()` if the
  task-id-or-status set changed.
- 2 s coalesce timer: drives lightweight re-paints of the count
  chip even when the watcher hasn't fired (e.g. status changes
  internal to a still-buffered write). Mirrors the bg-task
  status timer (`refreshBgTasksButton`).
- Watcher re-attach on atomic-rewrite. The new tracker's
  `rescan()` calls `m_watcher.addPath()` if the file exists
  but isn't watched, identical to the pattern in
  `claudebgtasks.cpp:91–95` — same code shape, separate
  member instance. Not aspirational reuse.

## 6. Status-bar widget

- **Placement:** `m_statusBar` permanent-widget area,
  immediately to the right of `m_bgTasksBtn` (the two
  Claude-Code surfaces sit adjacent).
- **Glyph:** clipboard / list icon at
  `assets/icons/tasklist.svg` (committed to the QRC). Unicode
  fallback if the SVG fails to load: `U+2630` (☰ TRIGRAM FOR
  HEAVEN — visually a list-of-rows glyph; renders in every
  monospace font on the project's tested distros).
- **Label:** `"<unfinished>/<total>"` where
  `unfinished = in_progress + pending` and `total =`
  list size. Single-number form picked over `1+2/5` for
  status-bar density. Example: `3/5`.
- **Hidden** in any of these cases (single rule — no
  flicker mitigation needed):
  - Focused tab has no transcript path (no `claude` running),
    or
  - The transcript has no plan events at all, or
  - The replayed plan list is empty (e.g. the assistant
    issued `TodoWrite([])` to clear the plan, OR every row
    was completed and Claude Code retained the empty state).
- **Tooltip** uses the user's verbatim status vocabulary:
  `"Claude Code task list — N tasks (X done, Y running,
  Z outstanding). Click to view."`. Internal status names
  stay JSON-canonical (`completed` / `in_progress` /
  `pending`); only the tooltip and dialog labels surface
  the user's wording.
- **Click:** emits `tasksClicked()` to MainWindow, which
  opens `ClaudeTaskListDialog`.

## 7. Dialog

- **Class:** `ClaudeTaskListDialog : public QDialog`.
- **Constructor:** `(ClaudeTaskListTracker *tracker,
  QString themeName, QWidget *parent)`. The constructor
  stores `themeName` and immediately calls a private
  `rebuild()` that populates the list — same pattern as
  `ClaudeBgTasksDialog` (which stores `m_themeName` and
  rebuilds on construction + on `tasksChanged`). No
  `applyTheme()` member; `rebuild()` consumes the stored
  theme name when it constructs colours.
- **Body:** scrollable list, one row per task in
  **parser-emitted order**. For Mode A (TodoWrite snapshot)
  that is the order of the most recent `input.todos` array;
  for Mode B (TaskCreate replay) it is creation order.
  **No stable identity is assumed across snapshots** — Mode
  A's TodoWrite *replaces* the list, so a logical task can
  appear at a different row index after a snapshot, and
  rows in an open dialog will jump on rebuild. This is
  expected and matches Claude Code's own TUI behaviour.
- **Row contents:**
  - Status icon: ☐ pending / ◐ in_progress / ✓ completed.
  - **Subject** (bold).
  - Description / `activeForm` (one line, truncated to
    ~200 chars with ellipsis).
- **Header:** `"<N> tasks — <Y> running, <Z> outstanding,
  <X> done"` (user vocabulary, mirrors the tooltip).
- **Live updates:** connect tracker's `tasksChanged()` to
  `rebuild()`. Same pattern as `ClaudeBgTasksDialog`.
- **Modality:** non-modal (matches `ClaudeBgTasksDialog` so
  the user can keep terminal interaction live). Spec
  invariant — neither `setModal(true)` nor
  `QDialogButtonBox` (both implicated in
  QTBUG-79126 — see `debug_wayland_modal_dialog` memory).
  Buttons, if any, are plain `QPushButton`s with explicit
  `clicked → close` connects.

### 7.1 Tracker lifetime

- The `ClaudeTaskListTracker` is **owned by
  `ClaudeStatusBarController`** (one tracker per controller,
  retargeted via `setTranscriptPath` on tab switch — same
  ownership model as `m_bgTasks`).
- The dialog holds a raw `ClaudeTaskListTracker *` that is
  guaranteed valid for the dialog's lifetime: the controller
  outlives MainWindow, and MainWindow parents the dialog,
  so dialog destruction precedes tracker destruction.
- On tab switch *while the dialog is open*: the controller
  retargets the tracker to the new tab's transcript path.
  The tracker emits `tasksChanged()` with the new (possibly
  empty) plan; the dialog's `rebuild()` re-renders with no
  rows. The dialog **does not** close itself on retarget — a
  user who opened it for tab A and then switched to tab B
  sees tab B's plan instead. Hide-on-empty applies to the
  status-bar chip only, not the dialog.

## 8. Out-of-scope (deliberate MVP limits)

- **No editing.** Read-only view; user cannot mark tasks
  done from the dialog. (Claude Code owns task state.)
- **No history.** Once a `TodoWrite([])` clears the list or
  the session ends, the dialog hides. We don't surface the
  prior plan.
- **No cross-session aggregation.** One transcript per tab.
- **No persistence.** Dialog state (last position) is not
  saved across sessions in this phase. (Could fold into
  ANTS-1150 Phase 2 if asked.)
- **No background-task overlap.** Background-tasks dialog
  (`ClaudeBgTasksDialog`) and task-list dialog stay
  independent — one tracks `Bash --run-in-background` /
  `Task --run-in-background` shells, the other tracks
  `TodoWrite` / `TaskCreate` plans. They could merge
  visually later but the MVP keeps them separate.

## 9. Invariants (for `tests/features/claude_task_list/`)

Test harness shape: a standalone C++ test linking only the
`src/*.cpp` objects it exercises (`claudetasklist.cpp`,
`claudetasklistdialog.cpp`), GUI-free where the assertion
allows. **Same shape as `tests/features/claude_bg_tasks/`**
(per `CLAUDE.md` test-harness conventions: the bg-tasks
fixtures use `add_executable` + `add_test` linking the source
under test, not pure source-grep).

Mode-A (TodoWrite snapshot) vs Mode-B (TaskCreate replay)
INVs are tagged so a future grep-and-fix pass can find each
mode's coverage without re-reading the file.

- **INV-1 (Mode A)** — Parser ingests synthetic JSONL with
  one `TodoWrite` event in the tail and produces N tasks
  matching `input.todos` array (status preserved). EOF-walk
  finds it without scanning the whole file.
- **INV-2 (Mode A)** — A second `TodoWrite` later in the
  transcript wins (most-recent semantics).
- **INV-3 (Mode B)** — Parser handles a `TaskCreate`
  tool_use followed by a paired `tool_result` and emits an
  entry with `status="pending"`. ID extracted from the
  paired result's body when present.
- **INV-4 (Mode B)** — Parser handles `TaskUpdate` flipping
  status on an existing entry; no-op when the `taskId` is
  unknown.
- **INV-5 (filter)** — Events with `isSidechain == true`
  are skipped — a synthetic JSONL with a sidechain
  `TaskCreate` produces zero plan rows.
- **INV-6 (filter)** — A `Task` tool_use whose `input` has
  `subagent_type` is treated as a subagent dispatch and
  filtered out (does not appear as a plan row).
- **INV-7 (lifecycle)** — `setTranscriptPath("")` clears
  state and emits `tasksChanged()` exactly once.
- **INV-8 (lifecycle)** — `setTranscriptPath(p)` for the
  same `p` twice is idempotent (no second emit).
- **INV-9 (widget)** — Status-bar widget is **hidden** when
  the parsed list is empty; **shown** when ≥ 1 task exists.
- **INV-10 (widget)** — Status-bar widget label matches
  `"<unfinished>/<total>"`.
- **INV-11 (dialog)** — Dialog body renders one row per
  task in parser-emitted order (test via QListWidget item
  count + first-row text match against the synthetic input).
- **INV-12 (dialog)** — Dialog rebuilds live on a
  `tasksChanged()` emit after `show()`; QListWidget item
  count updates within one event-loop turn (driven by
  `QCoreApplication::processEvents`).
- **INV-13 (dialog anti-regression)** — Dialog source does
  NOT contain `setModal(true)` AND does NOT use
  `QDialogButtonBox` (both Wayland-flake patterns). Source-
  grep check.

## 11. Acceptance

Spec is met when:

1. Running `claude` in tab A, issuing 3 `TaskCreate` calls
   (or one `TodoWrite` with 3 todos) where 1 is done, 1 is
   in_progress, 1 is pending, shows `2/3` on the status
   bar of tab A only. (Mode A and Mode B reach the same
   chip text.)
2. Clicking the chip opens a dialog listing all three with
   correct status icons + the user-vocab header
   (`X done, Y running, Z outstanding`).
3. Switching to tab B (no Claude session) hides the chip;
   switching back to A shows it again with the same count.
4. From inside Claude Code, calling `TaskUpdate` to flip
   the in_progress task to completed updates the chip to
   `1/3` within ≤ 2 s and the open dialog row icon flips
   to ✓.
5. Clearing the plan (every row → completed, OR
   `TodoWrite([])` in older Claude Code) hides the chip.
6. Spawning a subagent via `Task --subagent-type=...` does
   NOT add a row to the user's plan (filtered).
7. A subagent's own `TaskCreate` (which inlines as
   `isSidechain:true` in the parent transcript) does NOT
   appear in the parent's chip count.
8. All INVs in `tests/features/claude_task_list/` pass.
