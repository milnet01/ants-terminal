# ANTS-1158 — Claude Code task-list status-bar surface

> **Source:** user 2026-05-02 — *"When Claude Code has a task
> list, please add a button to the task bar that brings up a
> dialog that shows all the tasks. It must show the task status
> (completed, running, outstanding)."*
>
> **Lanes:** new `claudetasklist.{cpp,h}` +
> `claudetasklistdialog.{cpp,h}`; hooks into
> `claudestatuswidgets` (status-bar widget) and `MainWindow`
> (tab-switch retargeting).
> **Kind:** implement

## 1. Problem

A focused tab whose Claude Code session has an active task list
gives the user no way, from inside Ants Terminal, to see the
plan's contents or progress when they're looking at a sibling
tab or the assistant has collapsed the list to a summary.

## 2. Goal

A status-bar widget on the focused tab that, when a task list
exists, shows a count chip (e.g. `3/5` for in-progress + pending
out of total) and on click opens a dialog listing every task in
the current plan with its status:

- **completed** (✓)
- **in_progress** ("running" in user's wording — ◐)
- **pending** ("outstanding" in user's wording — ☐)

When no task list exists for the focused tab, the widget is
hidden.

## 3. Source of truth

Claude Code persists task state **only** in the active session
JSONL transcript at:

```
~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl
```

(`~/.claude/tasks/<session-uuid>/` exists but only holds `.lock`
and `.highwatermark` — no payloads. Verified 2026-05-02.)

### 3.1 Discriminating plan events from subagent dispatches

The Claude Code "Task" tool family overlaps two roles:

| Event shape | Role | Treat as plan event? |
|---|---|---|
| `name == "TaskCreate"`, `input` has `{subject, activeForm, ...}` | Add a row to the user-visible plan | **YES** |
| `name == "TaskUpdate"`, `input` has `{taskId, status, ...}` | Flip the status of a row | **YES** |
| `name == "TodoWrite"`, `input` has `{todos: [...]}` | Full snapshot of the plan (older Claude Code versions) | **YES — replaces list** |
| `name == "Task"` or `"Agent"`, `input` has `subagent_type` AND/OR `prompt` | Spawn a sub-agent (NOT a plan row) | **NO — filter out** |
| `name == "TaskList"` / `"TaskGet"` / `"TaskStop"` / `"TaskOutput"` | Read-only / control | **NO — ignore** |

Verified 2026-05-02 against the user's transcripts: TaskCreate
appears with the `{subject, description, activeForm}` shape —
that is the plan-add shape, distinct from the subagent-dispatch
shape (`prompt` + `subagent_type`). TodoWrite is absent from the
user's history but the parser supports it for forward-compat
with environments where it's used.

Additional filter: **events with `isSidechain == true` are
ignored**. Subagent turns inline their tool calls in the parent
transcript with that flag; including them would make a parent
chip flicker on subagent task progress that the user never
asked to see.

### 3.2 Replay algorithm

The plan list is reconstructed by walking events in two
modes:

**Mode A — TodoWrite-only (snapshot mode).** If the transcript
contains *any* TodoWrite event, the most recent one wins.
Walk **backwards from EOF** to find it (cheap since the
relevant event is usually near the tail), then return its
`input.todos` as the plan. Discards the entire TaskCreate /
TaskUpdate stream.

**Mode B — TaskCreate / TaskUpdate replay (incremental mode).**
Used when no TodoWrite was found. Walk forward from the start,
applying:

```
state = empty list
for each event in chronological order:
    if event.isSidechain: continue
    for c in event.message.content[]:
        if c.type != "tool_use": continue
        if name == "TaskCreate":
            id = scan_paired_tool_result(c.id)   // arrives async
            state.append({id, subject, status: "pending", ...})
        elif name == "TaskUpdate":
            t = state.find(input.taskId)
            if t: apply update fields onto t
return state
```

Notes:
- `TaskCreate`'s allocated ID arrives in the *paired*
  `tool_result` (matched via `tool_use_id`). Mirrors the
  bg-task launch/confirmation pattern at
  `claudebgtasks.cpp:222–243`.
- A `tool_result` for an unmatched `tool_use_id` is ignored
  (in-flight failure or concurrent agent).

### 3.3 Replay scope and bounds

- **One transcript** — focused tab's session, no cross-session
  merge.
- **Mode A: no tail cap.** Read the file in 64 KiB blocks from
  EOF, scan for `"name":"TodoWrite"`, parse only that line.
  Bounds work without losing the snapshot.
- **Mode B: 16 MiB tail cap** (mirrors `claudebgtasks.cpp:147`).
  TaskCreate / TaskUpdate are append-only, so the dropped head
  is older history.
- **GUI-thread only.** Parser is `static` for testability but
  production callers always invoke under the
  QFileSystemWatcher signal on the GUI thread. Same Qt-I/O
  reentrancy reason as `claudebgtasks.cpp:137`.

## 4. Per-tab scoping

Status-bar widget reflects the focused tab's task list. On
`onTabChanged`, retarget the tracker to the new tab's
`transcriptPath`. Path comes from
`ClaudeTabTracker::ShellEntry::transcriptPath` keyed on the
shell PID — same lookup the bg-task tracker uses.

## 5. Update cadence

- One `QFileSystemWatcher` on the active transcript path.
  `fileChanged` → re-parse → emit `tasksChanged()` iff the
  task-id-or-status set changed.
- 2 s coalesce timer drives lightweight re-paints when the
  watcher hasn't fired (still-buffered writes). Mirrors
  `refreshBgTasksButton`.
- Watcher re-attach on atomic-rewrite: `rescan()` calls
  `m_watcher.addPath()` if the file exists but isn't watched.
  Same code shape as `claudebgtasks.cpp:91–95`, separate
  instance.

## 6. Status-bar widget

- **Placement:** `m_statusBar` permanent-widget area, right of
  `m_bgTasksBtn`.
- **Glyph:** `assets/icons/tasklist.svg` (committed to QRC).
  Fallback: `U+2630` (☰).
- **Label:** `"<unfinished>/<total>"` where unfinished =
  in_progress + pending. Example: `3/5`.
- **Hidden** when the focused tab has no transcript, the
  transcript has no plan events, or the replayed plan is
  empty (single rule — no flicker mitigation).
- **Tooltip:** `"Claude Code task list — N tasks (X done,
  Y running, Z outstanding). Click to view."`. User's
  vocabulary on the surface; internal status names stay
  JSON-canonical (`completed` / `in_progress` / `pending`).
- **Click:** emits `tasksClicked()` → MainWindow opens
  `ClaudeTaskListDialog`. Left-click only — no middle / right
  bindings in MVP.

## 7. Dialog

- **Class:** `ClaudeTaskListDialog : public QDialog`.
- **Constructor:** `(ClaudeTaskListTracker *, QString
  themeName, QWidget *parent)`. Stores `themeName`, calls
  `rebuild()` immediately. No `applyTheme()` member —
  `rebuild()` consumes the stored name. Mirrors
  `ClaudeBgTasksDialog`.
- **Body:** scrollable list, one row per task in
  parser-emitted order (Mode A = `input.todos` array order,
  Mode B = creation order). Row identity is NOT stable
  across snapshots — Mode A's TodoWrite replaces the list,
  so rows can re-order on rebuild. Matches Claude Code's
  own TUI.
- **Row:** status icon (☐ pending / ◐ in_progress / ✓ completed),
  bold subject, one-line description (≤ 200 chars, ellipsis).
- **Header:** `"<N> tasks — <Y> running, <Z> outstanding,
  <X> done"`.
- **Live updates:** `tracker->tasksChanged` → `rebuild()`.
- **Modality:** non-modal. Neither `setModal(true)` nor
  `QDialogButtonBox` (both implicated in QTBUG-79126; see
  `debug_wayland_modal_dialog` memory). Buttons, if any, are
  plain `QPushButton`s with explicit `clicked → close`.

### 7.1 Tracker lifetime

`ClaudeTaskListTracker` is owned by
`ClaudeStatusBarController` (one tracker, retargeted via
`setTranscriptPath` on tab switch — same model as
`m_bgTasks`). The dialog's raw pointer is valid for its
lifetime: the controller outlives MainWindow which parents
the dialog. On retarget while the dialog is open, the
tracker emits `tasksChanged()` with the new plan and
`rebuild()` re-renders — the dialog does not close itself.
Hide-on-empty applies to the chip only.

## 8. Out-of-scope (MVP)

- No editing. Read-only — Claude Code owns task state.
- No history. Cleared / ended → chip hides; no surfacing of
  prior plans.
- No cross-session aggregation. One transcript per tab.
- No dialog-state persistence (last position). Future fold
  into ANTS-1150 Phase 2 if asked.
- No background-task overlap. `ClaudeBgTasksDialog` tracks
  `--run-in-background` shells; this tracks the plan list.
  Independent.

   on the focused tab only. Mode A and Mode B reach the
   same chip text.
2. Click → dialog with 3 rows + header `1 done, 1 running,
   1 outstanding`.
3. Switch to a non-Claude tab → chip hidden; switch back →
   chip restored with the same count.
4. `TaskUpdate` flipping in_progress → completed updates
   chip to `1/3` and open-dialog row to ✓ within ≤ 2 s.
5. Plan cleared (all completed, or `TodoWrite([])`) → chip
   hidden.
6. Subagent dispatch (`Task` with `subagent_type`) does NOT
   add a plan row.
7. Subagent's own `TaskCreate` (`isSidechain:true`) does
   NOT count in the parent's chip.
