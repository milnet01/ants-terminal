## 1. Problem

Claude Code's TUI surfaces an in-progress task list (the running
plan from `TodoWrite` / `TaskCreate` / `TaskUpdate`) inline in the
chat pane. When the user is *not* looking at the Claude Code TUI ŌĆö
e.g. they switched to a sibling tab, or the assistant collapsed
the list to a one-line summary ŌĆö there's no way from inside Ants
Terminal to see the current plan or its progress.

## 2. Goal

A status-bar widget on the focused tab that, when a task list
exists, shows a count chip (e.g. `3/5` for in-progress + pending
out of total) and on click opens a dialog listing every task in
the current plan with its status:

- **completed** (Ō£ō)
- **in_progress** ("running" in user's wording ŌĆö ŌŚÉ)
- **pending** ("outstanding" in user's wording ŌĆö ŌśÉ)

When no task list exists for the focused tab, the widget is
hidden.

## 3. Source of truth

Claude Code persists task state **only** in the active session
JSONL transcript at:

```
~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl
```

(`~/.claude/tasks/<session-uuid>/` exists but only holds `.lock`
and `.highwatermark` ŌĆö no payloads. Verified 2026-05-02.)

Three event kinds carry task data:

1. **`TodoWrite` tool_use** ŌĆö full snapshot. `input.todos = [{
   activeForm, content, status }, ...]`. The list at this event
   is canonical for the moment of the event.

2. **`TaskCreate` tool_use** ŌĆö incremental add. `input = { subject,
   description, activeForm }`. Tool result returns the new task ID.

3. **`TaskUpdate` tool_use** ŌĆö incremental status flip. `input = {
   taskId, status, ... }`.

`TaskList` / `TaskGet` / `TaskStop` / `TaskOutput` are read-only
and don't mutate state.

### 3.1 Replay algorithm

```
state = empty list
for each line in transcript JSONL (chronological):
    if event.message.content[].type == "tool_use":
        if name == "TodoWrite":
            state = parse(input.todos)            // full replace
        elif name == "TaskCreate":
            id = scan_paired_tool_result(uuid)    // ID arrives async
            state.append({id, subject, status: "pending", ...})
        elif name == "TaskUpdate":
            t = state.find(input.taskId)
            if t: apply update fields onto t
return state
```

Notes:

- `TaskCreate` returns its allocated ID via the paired
  `tool_result` event. The replay must look at the *paired*
  tool_result (matched via `tool_use_id`) to learn the ID.
  Mirrors the bg-task launch/confirmation pattern in
  `claudebgtasks.cpp:222ŌĆō243`.
- `TodoWrite` fully replaces the list ŌĆö historical
  `TaskCreate` IDs may no longer be present. That's intended.
- An event with `content[].type == "tool_result"` for an
  unmatched `tool_use_id` is ignored (in-flight failure or
  concurrent agent).

### 3.2 Replay scope

- One transcript (the focused tab's session). No cross-session
  merge ŌĆö different sessions produce independent task lists.
- Cap at **16 MiB** of transcript tail (mirrors `claudebgtasks.cpp:147`)
  so the parse remains bounded on long-running sessions. Drop
  the first (likely-truncated) line on truncation.
- Pure / static parser exposed for unit tests (mirrors
  `ClaudeBgTaskTracker::parseTranscript`).

## 4. Per-tab scoping

A user can run `claude` in multiple tabs concurrently. The
status-bar widget reflects the **focused** tab's task list. On
tab switch (`onTabChanged` already fires), retarget the tracker
to the new tab's `transcriptPath` (same pattern
`ClaudeBgTaskTracker::setTranscriptPath` uses).

The transcript path for a tab is already known ŌĆö
`ClaudeTabTracker::ShellEntry::transcriptPath` carries it.
Look up via `m_tracker` on the focused shell PID.

## 5. Update cadence

- One `QFileSystemWatcher` on the active transcript path. On
  `fileChanged` re-parse and emit `tasksChanged()` if the
  task-id-or-status set changed.
- 2 s coalesce timer: drives lightweight re-paints of the count
  chip even when the watcher hasn't fired (e.g. status changes
  internal to a still-buffered write). Mirrors the bg-task
  status timer (`refreshBgTasksButton`).
- Watcher re-attach on atomic-rewrite (Claude Code appends, but
  some editors do atomic rename ŌĆö the existing
  `ClaudeBgTaskTracker::rescan` re-add path covers this).

## 6. Status-bar widget

- Placement: `m_statusBar` permanent-widget area, **immediately
  to the right of `m_bgTasksBtn`** (so the two Claude-Code
  surfaces are adjacent).
- Glyph: clipboard / list icon at `~/.config/ants-terminal/
  themes/icons/tasklist.svg` (use a Unicode fallback if SVG
  loading fails ŌĆö mirrors the `m_bgTasksBtn` SVG path).
- Label: `"<in_progress>+<pending>/<total>"` ŌĆö e.g. `1+2/5`
  when 1 in_progress, 2 pending, 2 completed (5 total).
  Or simpler: `"<in_progress+pending>/<total>"` ŌĆö e.g. `3/5`.
  **Spec defaults to the simpler form `3/5`**; revisit if user
  asks for more granularity.
- Hidden when:
  - Focused tab has no transcript path.
  - Transcript has no task-list events at all.
  - Task list is empty after replay (e.g. the assistant cleared
    it via `TodoWrite([])`).
- Tooltip: `"Claude Code task list ŌĆö N tasks (X done, Y in
  progress, Z pending). Click to view."`
- Click: emits `tasksClicked()` to MainWindow, which opens
  `ClaudeTaskListDialog`.

## 7. Dialog

- Class: `ClaudeTaskListDialog : public QDialog`.
- Constructor: `(ClaudeTaskListTracker *, const QString &theme,
  QWidget *parent)`.
- Body: scrollable list, one row per task in **creation
  order** (or ŌĆö for `TodoWrite` snapshots ŌĆö the order Claude
  Code emitted in the array). Each row:
  - Status icon: ŌśÉ pending / ŌŚÉ in_progress / Ō£ō completed.
  - **Subject** (bold).
  - Description / `activeForm` (one line, truncated to ~200
    chars with ellipsis).
- Header: `"<N> tasks ŌĆö <Y> in progress, <Z> pending, <X> done"`.
- Updates live: connect tracker's `tasksChanged()` to a slot
  that rebuilds the body. Same pattern as
  `ClaudeBgTasksDialog` connect.
- Modality: non-modal (matches `ClaudeBgTasksDialog` so the
  user can keep terminal interaction live). Spec invariant:
  not `setModal(true)` (per `debug_wayland_modal_dialog`
  memory).
- Theme: same theme-as-QSS hook the rest of the dialogs use
  ŌĆö `applyTheme(theme)` on construction; respond to
  `MainWindow::themeChanged` if it lands while open.

## 8. Out-of-scope (deliberate MVP limits)

- **No editing.** Read-only view; user cannot mark tasks
  done from the dialog. (Claude Code owns task state.)
- **No history.** Once a `TodoWrite([])` clears the list or
  the session ends, the dialog hides. We don't surface the
  prior plan.
- **No cross-session aggregation.** One transcript per tab.
- **No persistence.** Dialog state (last position) is not
  saved across sessions in this phase. (Could fold into
  ANTS-1150 Phase 2 if asked.)
- **No background-task overlap.** Background-tasks dialog
  (`ClaudeBgTasksDialog`) and task-list dialog stay
  independent ŌĆö one tracks `Bash --run-in-background` /
  `Task --run-in-background` shells, the other tracks
  `TodoWrite` / `TaskCreate` plans. They could merge
  visually later but the MVP keeps them separate.

## 12. Open questions for sign-off

1. **Granular vs simple count chip.** `1+2/5` (in_progress +
   pending separately) vs `3/5` (combined). Spec defaults
   to `3/5`; user can override.
2. **Click target in the chip.** Click whole chip ŌåÆ open
   dialog (current spec). Alternative: middle-click pins
   the chip to always-visible (no current use case;
   skipping).
3. **Sort order in the dialog.** Creation order (current
   spec) vs status grouping (in_progress first, then
   pending, then completed). Creation order matches
   Claude Code's TUI; recommend keeping.
4. **Ghost retention.** When the focused-tab session ends
   (PTY dies), the transcript file persists but no live
   updates arrive. Spec: keep showing the last known
   state until the user explicitly switches tabs or the
   tracker is retargeted. Alternative: hide on session
   end. Recommend keeping (matches bg-tasks behaviour).
