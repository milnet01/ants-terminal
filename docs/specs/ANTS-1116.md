## What v1 is (post cold-eyes pass 1)

**One subcommand: `drift-check`.** That's it. `audit-run` is
deferred to v2 — see the architectural finding below.

## Why ANTS-1116 ships after ANTS-1117 (and v1 is this small)

ANTS-1117 lights up the highest-leverage path inside the running
GUI. ANTS-1116 covers off-GUI cases — Claude on a CI runner, or
a session where the GUI isn't open. v1's only job is to **prove
the binary shape works** (CMake gating, argv parsing, JSON
output, exit-code semantics) on the smallest possible surface
that's actually useful. That surface is `drift-check`.

## Scope (v1)

A new optional binary `ants-helper`, built only when CMake is
invoked with `-DANTS_ENABLE_HELPER_CLI=ON` (default OFF for
v1). One subcommand:

| Subcommand | Input (JSON via argv or stdin) | Output (JSON to stdout) |
|------------|--------------------------------|------------------------|
| `drift-check` | `{}` | `{"ok": true, "data": {"clean": true}}` on no-drift; `{"ok": true, "data": {"clean": false, "violations": [{"file","line","message"}], "raw": "...", "exit_code": N}}` on drift detected; `{"ok": false, "error": "...", "code": "..."}` on handler error. Drift detection is **not** an `ok: false` case — the handler ran successfully; the *result* is "drift exists." Callers switch on `data.clean` for the drift signal; the exit code below disambiguates clean / drift / handler-error for shell consumers. |

Same unified `{ok, data}` / `{ok, error, code}` shape as
ANTS-1117 — keeps both surfaces consistent.

## Why `audit-run` is NOT in v1 (architectural finding 16-1)

Original ROADMAP body proposed `audit-run` in v1. Cold-eyes
review caught: `audit-run` would call into `auditdialog.cpp`,
which transitively pulls in `Qt6::Widgets`. Linking that into a
"stateless CLI" binary inflates the binary by ~30 MB and
leaks GUI dependencies into a tool that is positioned as
GUI-free.

The fix is **not** "live with the dependency" — it's to extract
the audit logic into a GUI-free module first. That's its own
bullet: ANTS-1119 (audit-engine extraction). Once 1119 lands,
`audit-run` is straightforward to add as a v2 subcommand.

## Why other subcommands are NOT in v1

- **`id-allocate`** — Claude does this in one Bash call already
  (`echo $(($(cat .roadmap-counter)+1)) > .roadmap-counter`).
  Token saving is negligible.
- **`test-runner`** — `ctest --output-on-failure` is already
  human-parseable. Structured JSON has some value but isn't
  the highest-leverage gap.
- **`audit-fold-in`, `roadmap-status`, `changelog-flip`,
  `bump`, `spec-parse`, …** — twelve other subcommands listed
  in the original ROADMAP entry. All deferred until ANTS-1120
  measurement validates the shape.

## Out of scope

- **MCP server.** The original spec proposed a sibling
  `ants-mcp-server` binary that wraps `ants-helper` for
  Model-Context-Protocol stdio transport. Defer to v2.
- **`~/.claude/CLAUDE.md` global rule.** The original spec proposed
  a §8 "prefer local helpers" disposition. Defer until measurement
  validates the framing.
- **Auto-discovery on `claude` startup.** Defer.
- **Network exposure.** Out of scope forever; this is a strictly
  local helper.

## Acceptance criteria (INVs)

- **INV-1** Building with `-DANTS_ENABLE_HELPER_CLI=OFF` (the
  default) produces no `ants-helper` binary; flipping to ON adds
  exactly one new binary target. The GUI binary's link line and
  output size are unchanged either way.
- **INV-2** `ants-helper drift-check` exits **0** with body
  `{"ok": true, "data": {"clean": true}}` when
  `packaging/check-version-drift.sh` exits 0 against a fresh tree.
- **INV-3** `ants-helper drift-check` exits **3** with body
  `{"ok": true, "data": {"clean": false, "violations": [...],
  "raw": "...", "exit_code": <script-exit-code>}}` (non-empty
  `violations`) when the drift script reports any drift on stdout.
  Drift detection is a *result*, not a handler error — `ok` stays
  `true`; the exit code (3) disambiguates "clean" (0) from "drift"
  (3) from "handler error" (1) for shell consumers. The wrapper
  trusts the script's exit code as the drift signal — only zero is
  "clean."
- **INV-3a** Response fields `raw` (string — the script's full
  stdout) and `exit_code` (number — the script's exit code) are
  always present in the drift-detected payload. Callers can replay
  the script's output untruncated via `raw`.
- **INV-4** Behaviour outside this repo: when CWD has no
  `packaging/check-version-drift.sh`, the handler returns
  `{"ok": false, "error": "drift script not found at <path>",
  "code": "missing_script"}` and exits **1**. No crash, no
  panic, no traceback.
- **INV-5** Behaviour when `bash` is not in `PATH`: handler
  returns `{"ok": false, "error": "bash unavailable", "code":
  "missing_bash"}` and exits **1**.
- **INV-6** Behaviour when the script is signal-killed
  (SIGTERM/SIGKILL): handler returns `{"ok": false, "error":
  "drift script killed by signal N", "code":
  "script_signaled"}` and exits **1**.
- **INV-7** All output is single-line UTF-8 JSON suitable for
  pipe consumption (no embedded literal newlines outside
  string values, no trailing whitespace, no BOM).
- **INV-8** Exit codes are stable: `0` = success (handler ran +
  result was "clean"), `1` = handler error or runtime failure,
  `2` = usage error (unknown subcommand, malformed JSON input),
  `3` = drift detected (handler ran fine; result is "not clean").
  `3` is reserved for `drift-check`; future subcommands may
  re-use it for their own "result is non-success but ran fine"
  semantics, documented per-subcommand.
- **INV-9** `ants-helper --help` prints the subcommand list and
  exits 0; `ants-helper unknown-cmd` prints a one-line error to
  stderr and exits 2.
- **INV-10** Stdin handling: when argv lacks a JSON arg or argv
  is `-`, the handler reads stdin until EOF; empty stdin is
  treated as `{}`. Behaviour is byte-identical between
  `ants-helper drift-check '{}'` and `echo '{}' | ants-helper
  drift-check`.

## Test plan

`tests/features/local_subagent_framework/`. The test links
`ants-helper.cpp` directly and calls the pure handler function
(`AntsHelper::driftCheck`) against synthetic inputs — exercises
the handler without spawning a subprocess, keeping CI fast. (The
v2 surface adds `AntsHelper::auditRun` once ANTS-1119 lands; that
test is out of v1 scope.)

A separate, smaller harness exercises the binary end-to-end via
`QProcess` to catch CLI-dispatcher regressions (argv parsing,
stdin handling, exit codes).
