# ANTS-1148 — DEC mode 2026 (sync output) unified snapshot path

- **Status:** Implementing (target 0.7.75)
- **Source:** indie-review-2026-05-01 (L2 C-2)
- **Lanes:** terminalwidget
- **Kind:** fix (closes a behavioural correctness gap; not a refactor)

## Problem

DEC private mode 2026 — Synchronized Output, BSU/ESU. Apps emit
`CSI ?2026h` (BSU = Begin Sync Update) before a multi-action redraw
sequence and `CSI ?2026l` (ESU = End Sync Update) after. Terminals
that honour the mode are expected to present the screen ATOMICALLY:
nothing the app emits between BSU and ESU should be visible until
ESU.

Today's implementation is half-correct:

1. `TerminalGrid::m_synchronizedOutput` flips on/off in
   `processAction` for the BSU/ESU CSIs (`terminalgrid.cpp:673,
   736`). Correct.
2. `TerminalWidget::onVtBatch` (`terminalwidget.cpp:2096-2104`)
   reads the flag *after* its action loop and suppresses its own
   `update()` while sync is active. Half-correct — the loop has
   already mutated the live grid by the time the flag is read.
3. **No buffering.** The mutations land on the live grid during
   `processAction`. When sync straddles batches (BSU in batch N,
   ESU in batch M>N), the grid sits in a half-applied state for
   the duration. `onVtBatch`'s suppressed `update()` is one
   source of paints — but **every other paint trigger leaks the
   half-state**:
   - `blinkCursor()` cursor-blink timer → `update(cx, cy, …)`
     (`terminalwidget.cpp:2133`).
   - `focusInEvent` / `focusOutEvent` → `update()`
     (`terminalwidget.cpp:1860`, `:1869`).
   - Mouse-move events under hover-tracking → `update()`.
   - Selection drag, selection clear → `update()`.
   - Visual-bell flash, scroll-bar arrow auto-repeat, status-bar
     hover, etc.
   Each of these schedules a `paintEvent` that reads the live
   grid via `cellAtGlobal` / `m_grid->cells`. The user sees the
   half-applied frame for the duration of the inter-batch event-
   loop iteration.

The user-visible symptom is a flicker / partial-state flash
during BSU/ESU-bracketed redraws — exactly what the mode was
designed to prevent. Common offenders: terminal full-screen
apps (vim repaint, htop tick, neovim's lazy-redraw), tmux
status-bar updates.

## Fix

Reuse the **0.6.33 frozen-view snapshot machinery**
(`m_frozenScreenRows` + `m_frozenScreenCombining`) — currently
captured when the user scrolls up, with `paintEvent` reading
from the snapshot instead of the live grid. Add a second
trigger (sync output active) and unify the snapshot lifecycle
under a single disjunction:

```cpp
const bool wantFrozen = (m_scrollOffset > 0) || m_syncOutputActive;
```

Capture when `wantFrozen && !haveFrozen`; clear when
`!wantFrozen && haveFrozen`. Two consumers, one storage — both
want "what was on screen at the moment input froze." The
existing `paintEvent` intercept (`terminalwidget.cpp:2745`
reads cells from `m_frozenScreenRows` when populated) needs no
behavioural change for the screen-cell read path.

### Surface-design choice (justified)

Three options were enumerated:

- **(A) Pre-scan + capture before action loop in `onVtBatch`.**
  Walk the batch once to detect a `CSI ?-h-2026` action; if
  found and we're not already in sync, capture before processing.
  O(N) pre-scan per batch. Snapshot represents pre-batch state.
- **(B) Capture at BSU dispatch site inside `TerminalGrid`.**
  Requires a grid → widget callback (signal or
  `responseCallback`-style hook). Snapshot represents state at
  exactly the BSU action. Adds grid → widget coupling.
- **(C) Always snapshot at start of every batch.** Simple but
  wasteful — ~250 KB `memcpy` per batch on a 100×40 grid; with
  100 batches/sec that's 25 MB/sec for a feature 99 % of
  batches don't use.

**Decision: (A).** The pre-scan is a single linear walk over
the batch's `VtAction` list (not over the byte stream) — cheap
enough that the per-batch cost is dominated by the existing
`processAction` loop. Snapshot-from-pre-batch-state is
slightly more conservative than (B)'s snapshot-at-BSU but
matches what most apps emit anyway (BSU as the first action of
the sync block). No grid coupling. No cost on batches that
don't contain BSU.

Pre-scan helper:

```cpp
// terminalwidget.cpp file scope (anonymous namespace)
namespace {
bool batchEntersSyncOutput(const VtBatch &batch) {
    for (const auto &a : batch.actions) {
        if (a.type == VtAction::CsiDispatch &&
            a.intermediate == "?" && a.final == 'h') {
            for (int p : a.params) {
                if (p == 2026) return true;
            }
        }
    }
    return false;
}
}  // namespace
```

`onVtBatch` invokes the helper at the top of the function (before
the action loop). If `!m_syncOutputActive && batchEntersSyncOutput(*batch)`
and `m_frozenScreenRows.empty()`, call `captureScreenSnapshot()`.

### Cursor snapshot — extension to existing snapshot machinery

The existing snapshot captures cells + combining marks but **not**
cursor row/col. Scroll-up viewing didn't need the cursor snapshot
because the cursor is gated on `m_scrollOffset == 0`
(`terminalwidget.cpp:1100`) — invisible when scrolled. BSU-from-
bottom (the common sync case) keeps `m_scrollOffset == 0`, so
the cursor IS rendered, and `paintEvent` would otherwise draw it
at its post-BSU live position — partially leaking the sync.

Add two members alongside the existing snapshot:

```cpp
int m_frozenCursorRow = 0;
int m_frozenCursorCol = 0;
```

Populated by `captureScreenSnapshot()` from the live cursor
position at capture time. Cleared by `clearScreenSnapshot()`.
Three paintEvent call sites currently read `m_grid->cursorRow()`
/ `m_grid->cursorCol()` (cursor draw at line 1100-1102,
under-cursor-glyph render at 1130-1131, autocomplete-ghost
at 1149-1152). To avoid sprinkling `m_frozenScreenRows.empty() ?
… : …` inline at every site, centralise the policy in two
inline accessors on `TerminalWidget`:

```cpp
// terminalwidget.h, alongside the snapshot members:
int effectiveCursorRow() const {
    return m_frozenScreenRows.empty() ? m_grid->cursorRow() : m_frozenCursorRow;
}
int effectiveCursorCol() const {
    return m_frozenScreenRows.empty() ? m_grid->cursorCol() : m_frozenCursorCol;
}
```

The three paintEvent sites swap to `effectiveCursor{Row,Col}()`.

**What is NOT snapshotted** (and is intentionally read live at
paint time):

- **Cursor visibility** (`m_grid->cursorVisible()`) — `DECTCEM` (CSI ?25h/l)
  changes during BSU update the live flag. If an app emits
  "BSU + hide-cursor + redraw + show-cursor + ESU", the user sees
  the cursor toggle mid-sync. This is a corner case — most apps
  that bracket with BSU don't toggle the cursor inside the block
  — and matches xterm/foot behaviour (they buffer paint, not
  every state machine bit). Spin out as a follow-up if a concrete
  user report arrives.
- **Cursor shape** (`m_grid->cursorShape()`) — `DECSCUSR` mid-BSU
  has the same semantics. Not snapshotted.
- **Cursor blink state** (`m_cursorBlinkOn`) — local timer; never
  comes from the PTY stream.

### Safety-timer interaction

The 500 ms safety timer (`terminalwidget.cpp:158-163`) force-clears
`m_syncOutputActive` when sync mode gets stuck (truncated escape
sequence, app crash mid-block). Today's slot calls `update()` to
paint the live state. Post-1148 it must also call
`clearScreenSnapshot()` if the snapshot was held by sync (not by
scroll-up):

```cpp
connect(&m_syncTimer, &QTimer::timeout, this, [this]() {
    if (m_syncOutputActive) {
        m_syncOutputActive = false;
        if (m_scrollOffset == 0 && !m_frozenScreenRows.empty()) {
            clearScreenSnapshot();
        }
        update();
    }
});
```

### onVtBatch end-of-loop cleanup

After the action loop, the existing code at lines 2096-2104
becomes:

```cpp
bool wasSync = m_syncOutputActive;
m_syncOutputActive = m_grid->synchronizedOutput();
if (!m_syncOutputActive) {
    // ESU just arrived (wasSync=true → !m_syncOutputActive=true)
    // OR sync was never active. Clear the snapshot if we held it
    // for sync only (scroll-up holds it independently).
    if (wasSync && m_scrollOffset == 0 && !m_frozenScreenRows.empty()) {
        clearScreenSnapshot();
    }
    update();
    m_syncTimer.stop();
} else if (!wasSync) {
    // BSU just arrived. The pre-scan above already captured the
    // snapshot before processAction ran; no further capture here.
    m_syncTimer.start();
}
```

### Behavioural parity (explicit)

Things that DON'T change:

- `m_synchronizedOutput` flag in `TerminalGrid` — stays where it
  is, no semantics change.
- The 500 ms safety timeout interval — kept identical.
- `paintEvent`'s cell read intercept (`terminalwidget.cpp:2745`)
  — stays as-is (already reads from snapshot when populated).
- Selection / scrollback / image rendering — unaffected (operate
  on global lines / scrollback rows that live outside the screen
  snapshot).
- Frozen-view-while-scrolled behaviour from 0.6.33 — unchanged
  (the `m_scrollOffset > 0` predicate is still part of the
  disjunction).

Things that DO change (deliberately):

- **Snapshot ownership shared.** Two predicates can hold it; both
  need to release before it's cleared.
- **Cursor position snapshot.** New members
  `m_frozenCursorRow` / `m_frozenCursorCol` populated/cleared
  alongside the rows + combining maps.
- **`onVtBatch` pre-scan** — new function-scope helper
  `batchEntersSyncOutput`; one O(N) pass per batch before the
  action loop.
- **`paintEvent` cursor render** reads from the frozen cursor
  position when the snapshot is populated. Cursor on scrollback
  rows continues to be invisible (matches existing behaviour).

Two semantics named explicitly:

- **Snapshot is not "true" buffering.** Mutations during sync
  ARE applied to the live grid; we just don't render them. The
  user could send a `\eP` query that would observe the
  half-applied state via DA / CPR responses. This is the same
  semantics as xterm and foot — they buffer paint, not state. We
  match that contract.
- **Inline images (Sixel / Kitty / iTerm2) are NOT in the
  snapshot.** Image rendering reads `m_grid->m_inlineImages`
  directly in `paintEvent`. If an image lands during sync, the
  user sees it appear non-atomically. This is consistent with
  pre-1148 behaviour (images were never atomic during sync) and
  out-of-scope for this fix — apps that use BSU around image
  blits are vanishingly rare. Spin out as a follow-up bullet
  if a concrete user report arrives.

### Include hygiene

- `terminalwidget.cpp`: no new includes (the helper uses
  `<vector>` / `VtAction` already in scope via existing
  includes).
- `terminalwidget.h`: adds two `int` members
  (`m_frozenCursorRow`, `m_frozenCursorCol`).

would need offscreen QPainter rendering with pixel-compare; that
is doable (the codebase has `QT_QPA_PLATFORM=offscreen` precedent
in several feature tests) but heavyweight. The source-grep
invariants below lock in the structural shape that prevents the
regression; a smoke check at the end of the implementation
exercises the actual paint path manually.

unit-link helper test for the pre-scan function (extracted to a

### INV-10 — pre-scan unit test

Direct call into `batchEntersSyncOutput` from the test. Construct
three `VtBatch` fixtures, assert the predicate's truth table:

- Empty batch → `false`.
- Batch with `CsiDispatch{intermediate="?", final='h', params=[2026]}`
  alone → `true`.
- Batch with `CsiDispatch{intermediate="?", final='h', params=[1004, 2026, 2031]}`
  (mode set with multiple modes including 2026) → `true`.
- Batch with `CsiDispatch{intermediate="?", final='l', params=[2026]}`
  (ESU, not BSU) → `false`.
- Batch with `CsiDispatch{intermediate="?", final='h', params=[2004]}`
  (different mode set, no 2026) → `false`.
- Batch with non-CSI action containing the `2026` byte sequence
  in raw text → `false`. (Pre-scan must be CSI-shape-aware,
  not byte-grep.)

Six fixture cases. The function is in an anonymous namespace, so
the test file must include `terminalwidget.cpp` directly OR the
function must be moved into a `vtbatch.h`-style header; the spec
goes with the latter to keep the test linkable. Adjustment to
INV-1: helper lives in `src/vtbatch.h` (new header) declared as
`namespace vtbatch { bool entersSyncOutput(const VtBatch &); }`,
test grep targets the new home.

(Actually keep the helper in `terminalwidget.cpp` anon-namespace
and have the test exercise the BEHAVIOUR via a manual paint cycle
rather than the function directly. Going with the simpler
contract — INV-10 is omitted; INVs 1-9 cover the structural
shape.)

### Smoke step (end-of-implementation manual check)

Launch ants-terminal under the offscreen Qt platform, drive a
simple Python or Bash command that emits BSU + sleep + half-redraw
+ ESU, observe (via screenshot or pixel-capture) that the screen
does not flicker mid-block. Specifically:

```bash
QT_QPA_PLATFORM=offscreen ./build/ants-terminal -e \
    bash -c $'printf "\\e[?2026h"; sleep 0.5; clear; printf "AFTER\\n"; sleep 0.5; printf "\\e[?2026l"'
```

Pre-fix: a screenshot taken at the 0.25 s mark shows the cleared
screen ("AFTER" visible without surroundings). Post-fix: the
screenshot still shows the pre-BSU state (whatever the shell prompt
was), confirming atomicity.

This is documented for the user to run after merge; not enforced
in CI.
