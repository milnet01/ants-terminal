# ANTS-1119 — Extract audit logic into a GUI-free engine module

- **Status:** Draft (awaiting user sign-off)
- **Owner:** Project lead
- **New bullet introduced by [ADR-0002](../decisions/0002-cold-eyes-companion-cleanup.md) cold-eyes pass 1.**
- **Source:** cold-eyes review 2026-04-30

## Why this bullet exists

`auditdialog.cpp` is ~5000 lines and interleaves three concerns:

1. **Engine** — orchestrating detector subprocesses (cppcheck,
   clazy, semgrep, ruff, …), parsing each detector's output
   into `Finding` records, applying the dedup/baseline-diff
   pipeline, calibrating against `audithygiene` rules.
2. **Triage** — confidence scoring, allowlist matching,
   severity shifts, inline-suppression parsing.
3. **Presentation** — `QTextBrowser` rendering, the SARIF /
   HTML export emitters, the dialog's UI controls and signal
   wiring.

For the GUI user, mixing them is fine. For any non-GUI consumer
(CI runner, `ants-helper audit-run`, future MCP server), the
engine + triage halves are the only parts wanted, and pulling
in the presentation half drags the entire `Qt6::Widgets`
dependency tree into a binary that doesn't need it.

ANTS-1119 is the prep work that unblocks every non-GUI audit
consumer: ANTS-1116 v2's `audit-run`, any CI lane that wants
structured findings without spawning a window server, the
ANTS-1117 v2 `audit-run` IPC verb.

## Scope

A pure refactor — no behavioural change visible to existing
GUI users. Two new files:

- **`src/auditengine.{h,cpp}`** — the engine + triage halves,
  depending only on `Qt6::Core` (no `Widgets`, no `Gui`).
  Surface:
  - `AuditEngine::Config` — input config (which detectors to
    run, scope paths, allowlist path).
  - `AuditEngine::run()` — synchronous entry point, returns
    `QList<Finding>`.
  - `AuditEngine::Finding` — POD-ish struct (file, line,
    rule, severity, message, confidence, snippet, blame).
  - Detector adapters as a flat list of free functions inside
    the engine TU; no inheritance hierarchy until ANTS-1116
    v2 demands one.

`auditdialog.cpp` becomes the presentation layer over the
engine:

- Constructs `AuditEngine::Config` from the dialog's UI state.
- Calls `AuditEngine::run()` (off-thread via existing worker
  pattern).
- Renders the returned `QList<Finding>` through the existing
  `QTextBrowser` HTML pipeline.
- The SARIF/HTML export emitters move into the engine module
  (they're pure data → text transforms; no widget deps).

## Out of scope

- **Behaviour change.** This is a pure refactor. Existing audit
  output for any input must be byte-identical pre-/post-refactor.
- **API stability beyond v1.** The `AuditEngine` surface is
  internal — ANTS-1116 v2 will be the first consumer outside
  the dialog and may force surface changes. The spec doesn't
  promise external API stability yet.
- **New detectors.** This bullet ships the same detectors
  `auditdialog.cpp` runs today. Adding new detectors is its
  own work.
- **Performance.** This bullet doesn't aim to make audits
  faster; ANTS-1115 (the per-subsystem perf-hot-path sweep —
  see ROADMAP.md `### ⚡ Performance — hot-path sweep`) is the
  place for that.

## Acceptance criteria (INVs)

- **INV-1** `src/auditengine.{h,cpp}` build successfully against
  **only `Qt6::Core`** — verified by a feature-test target whose
  link line includes `auditengine.cpp` but not `Qt6::Widgets`
  or `Qt6::Gui`.
- **INV-2** `auditdialog.cpp`'s LOC drops by at least 30 %; the
  engine + triage code lives in `auditengine.cpp`.
- **INV-3** Running the full audit pipeline on this repo
  pre-refactor and post-refactor produces **byte-identical**
  finding lists (verified by hashing the sorted finding-stream
  serialisation).
- **INV-4** Running the SARIF export pre-/post-refactor on the
  same finding list produces byte-identical SARIF documents.
- **INV-5** Running the HTML export pre-/post-refactor on the
  same finding list produces byte-identical HTML.
- **INV-6** No new public headers leak `Qt6::Widgets` /
  `Qt6::Gui` types into the engine surface (`AuditEngine::*`
  forward-declares only `Qt6::Core` types).
- **INV-7** Existing audit-related feature tests
  (`tests/features/audit_*`) pass unchanged. Any test that
  previously linked `auditdialog.cpp` to exercise engine
  behaviour switches to linking `auditengine.cpp` directly.

## Test plan

- **`tests/features/audit_engine_extraction/`** — new feature
  test:
  - Source-grep INVs: `auditengine.h` exists and uses only
    `Qt6::Core` includes; no `QWidget` / `QDialog` / `QPainter`
    references.
  - Behavioural INV: link `auditengine.cpp` directly, run the
    pipeline against a synthetic source corpus, compare the
    finding stream byte-by-byte against a snapshot recorded
    pre-refactor.
- The byte-identical SARIF / HTML INVs use snapshot files
  recorded just before the refactor commit.

4. `tests/features/audit_engine_extraction/`

## Dependencies

- **Blocks:** ANTS-1116 v2 (`audit-run` subcommand) and
  ANTS-1117 v2 (`audit-run` IPC verb).
- **Blocked by:** none.

## Open questions for sign-off

1. The 30 % LOC drop is a soft target; if the actual extraction
   yields less because too much was already separable, do we
   flag that as a finding or accept it?
2. Should the engine be in `src/auditengine.{h,cpp}` (alongside
   the rest of `src/`) or in a new subdirectory like
   `src/audit/`? (My recommendation: flat for v1; promote to a
   subdirectory only when it grows beyond two source files.)
